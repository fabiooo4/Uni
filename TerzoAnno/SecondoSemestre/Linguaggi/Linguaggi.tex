\section{Linguaggi di programmazione}
Un linguaggio interpretato è un linguaggio di programmazione in cui le istruzioni vengono
eseguite direttamente, senza la necessità di essere compilate in un formato eseguibile.
Un interprete è rappresentato da un modello di esecuzione, cioè un \textbf{control
flow graph} (CFG) che rappresenta il flusso di controllo del programma.

I programmi si possono interpretare in diversi modi:
\begin{itemize}
  \item \textbf{Interpretazione diretta}: l'interprete esegue direttamente le istruzioni del
    programma e segue il flusso di controllo rappresentato dal CFG restituendo i risultati.

  \item \textbf{Interpretazione del segno}: l'interprete esegue le istruzioni del programma
    ma invece di restituire i risultati, restituisce un segno che rappresenta il risultato.
    Questo è utile quando non si riesce a restituire un risultato concreto, ad esempio quando
    un algoritmo non termina.

  \item \textbf{Analisi statica}: l'interprete analizza il programma senza eseguirlo, ad
    esempio per verificare la correttezza del codice o per ottimizzare le prestazioni.
\end{itemize}


\subsection{Obiettivo dei linguaggi di programmazione}
\noindent
L'obiettivo è creare un modello formale che riesca ad eseguire i costrutti necessari
per eseguire un programma. I componenti principali sono:
\begin{itemize}
  \item \textbf{Dati}
  \item \textbf{Algoritmi}: Calcolo da eseguire sui dati, cioè calcola funzioni.
\end{itemize}

\vspace{1em}
\noindent

\begin{definition}[Algoritmo]
    Un algoritmo è una sequenza di passi primitivi finita. L'algoritmo è descritto
    attraverso un programma in un linguaggio di programmazione.
\end{definition}

\vspace{1em}
\noindent

\begin{definition}[Linguaggio di programmazione]
  Un linguaggio di programmazione è uno strumento formale per descrivere algoritmi
  applicati ai dati.
\end{definition}

\vspace{1em}
\noindent

\begin{definition}[Programma]
  Il programma è la realizzazione nel linguaggio di programmazione di un algoritmo
  (concetto astratto).
  È una rappresentazione sintattica, cioè rispetta delle regole sintattiche del
  linguaggio di programmazione, finita di comportamenti potenzialmente infiniti.
  I comportamenti potenzialmente infiniti sono funzioni calcolate dall'algoritmo
  rappresentato (o implementato) dal programma:
  \[
    f: Dom \to Codom \cup \{\uparrow\}
  \] 

  \noindent
  Il linguaggio di programmazione deve considerare strumenti per descrivere
  \textbf{metodi di calcolo}
\end{definition}

\noindent
Alcuni esempi di linguaggi sono:
\begin{itemize}
  \item \textbf{Linguaggio matematico}: notazione rigorosa per rappresentare funzioni.
    Questo linguaggio non è adatto per descrivere algoritmi perchè non ha strumenti
    che descrivono \textbf{come} calcolare funzioni.

    \begin{itemize}
      \item Paradigma funzionale: nascono dal linguaggio matematico per adattarlo
        alla descrizione di algoritmi. Derivano dal lambda calcolo e alcune implementazioni
        sono Haskell o Lisp.
    \end{itemize}


  \item \textbf{Linguaggi logici}: notazione rigorosa con regole-assiomi che permettono
    di specificare un processo di calcolo attraverso regole di derivazione. Il
    calcolo è rappresentato da un processo di dimostrazione. Manca la descrizione
    di calcoli potenzialmente infiniti.

    \begin{itemize}
      \item Paradigma logico: nascono dal linguaggio logico per adattarlo alla descrizione di
        algoritmi. Derivano dal calcolo dei predicati e un esempio di implementazione è
        Prolog.
    \end{itemize}

  \item \textbf{Linguaggi di programmazione}: notazione rigorosa, tramite grammatiche CF,
    con:
    \begin{itemize}
      \item \textbf{Regole}: descrivono la semantica delle istruzioni
      \item \textbf{Istruzioni}: permettono di descrivere i passi di calcolo
    \end{itemize}
\end{itemize}

\begin{example}
  Alcuni esempi di applicazioni dei linguaggi di programmazione sono:
  \begin{itemize}
    \item Applicazioni scientifiche: Fortran, ecc...
    \item Applicazioni economiche: Cobol, ecc...
    \item Applicazioni AI: Lisp, ecc...
    \item Applicazioni sistemi: C, ecc...
    \item Applicazioni web: JavaScript, Java, ecc...
  \end{itemize}
\end{example}

\subsection{Aspetti di progettazione}
Gli aspetti per scegliere un linguaggio di programmazione sono:
\begin{itemize}
  \item \textbf{Readability} (Leggibilità): quanto è facile leggere e comprendere i
    programmi, cioè:
    \begin{itemize}
      \item Sintassi chiara
      \item Nessuna ambiguità
      \item Facilità di analisi del codice
    \end{itemize}

  \item \textbf{Writability} (Facilità di scrittura): quanto è facile o agevole scrivere
    programmi.

  \item \textbf{Reliability} (Affidabilità): quanto il linguaggio è conforme alle specifiche
    e quanto riduce l'introduzione di errori di programmazione.

  \item \textbf{Cost} (Costo):
    \begin{itemize}
      \item Costi di efficienza
      \item Costi di apprendimento
      \item Costi di utilizzo in modo corretto
    \end{itemize}
\end{itemize}

\subsubsection{Tipi di linguaggi}
I linguaggi di programmazione si possono classificare come:
\begin{itemize}
  \item \textbf{Linguaggi di basso livello}: sono linguaggi vicini alla macchina fisica,
    come ad esempio l'assembly o il linguaggio macchina.

  \item \textbf{Linguaggi di alto livello}: sono linguaggi più vicini al linguaggio umano,
    come ad esempio:
    \begin{itemize}
      \item Linguaggi funzionali
      \item Linguaggi imperativi
      \item Linguaggi logici
    \end{itemize}

    \vspace{1em}
    \noindent
    I linguaggi si possono ulteriormente dividere in:
    \begin{itemize}
      \item \textbf{Linguaggi puri}: 
        I linguaggi imperativi e alcuni linguaggi funzionali sono detti \textbf{funzionali impuri}
        perchè permettono di descrivere algoritmi con variabili e assegnamenti che
        possono ricevere degli "effetti collaterali" (side effects) cioè modifiche
        dello stato del programma. Quindi la variabile è un'astrazione della cella di memoria,
        di conseguenza lo stesso nome in punti diversi può avere valori diversi.

      \item \textbf{Linguaggi puri}:
        I linguaggi logici e alcuni linguaggi funzionali sono detti \textbf{funzionali puri}
        perchè le variabili sono matematiche e quindi \textbf{immutabili}, cioè non possono
        essere modificate dopo essere state assegnate. In questo modo si evitano gli
        effetti collaterali.
    \end{itemize}

    \vspace{1em}
    \noindent
    I linguaggi di alto livello si possono dividere in base al paradigma di programmazione:
    \begin{itemize}
      \item \textbf{Object Oriented}: linguaggi che permettono di strutturare il codice
        in oggetti con attributi e metodi.

      \item \textbf{Procedurali}: linguaggi che permettono di strutturare il codice in procedure
        o funzioni.

      \item \textbf{Strutturati}: linguaggi che permettono di strutturare il codice in blocchi di istruzioni
        con costrutti di controllo come if, while, for, ecc...

      \item \textbf{Dichiarativi}: linguaggi che si basano sulla descrizione del problema
        piuttosto che sulla descrizione del processo di calcolo.
    \end{itemize}
\end{itemize}

\subsection{Implementare un linguaggio di programmazione}
Le caratteristiche che hanno effetto sulla progettazione di un linguaggio di programmazione
sono:
\begin{itemize}
  \item L'architettura della macchina su cui è implementato
  \item Le metodologie di progettazione di programmi, cioè i paradigmi di programmazione
\end{itemize}

\section{Esecuzione dei linguaggi}
L'obiettivo dei linguaggi è \textbf{eseguire i programmi}.
Per poter eseguire un programma serve un altro programma che lo esegue che è detto
\textbf{macchina astratta} per il linguaggio di programmazione.

\begin{definition}[Macchina astratta]
  Dato un linguaggio di programmazione \( L \), la macchina astratta per \( L \)
  \( M_L \) è un insieme di \textbf{strutture dati} e \textbf{algoritmi} che permettono
  di memoriazzare e eseguire programmi scritti in \( L \).

  \vspace{1em}
  \noindent
  Intuitivamente è la macchina il cui linguaggio è esattamente \( L \).

  \begin{itemize}
    \item Per ogni macchina astratta \( M_L \) esiste un unico linguaggio \( L \) 
      che è il linguaggio riconosciuto ed eseguito da \( M_L \) 
    \item Per ogni linguaggio di programmazione \( L \) possono esistere
      infinite macchine astratte per \( L \) 
  \end{itemize}
\end{definition}

\begin{example}
  La macchina fisica è la \textbf{macchina astratta} per il linguaggio macchina.
  Mentre tutte le macchine astratte per qualsiasi altro linguaggio di programmazione
  di alto livello
  sono software, quindi altri programmi che traducono le istruzioni del linguaggio di programmazione
  in un'istruzione in un livello più basso.
\end{example}

\begin{definition}[Linguaggio macchina]
  Data una macchina \( M \), il linguaggio macchina di \( M \) \( L_M \) è il linguaggio
  che ha come stringhe quelle direttamente interpretabili da \( M \).

  \vspace{1em}
  \noindent
  Se si ha una pila di macchine astratte \( \{M_{i-1}, M_i, M_{i+1}, M_{i+2}\} \)
  la macchina \( M_i \) ha il suo linguaggio macchina \( L_{i} \) che utilizza il
  linguaggio macchina \( L_{i-1} \) della macchina \( M_{i-1} \) per eseguire i
  programmi scritti in \( L_i \). Inoltre il linguaggio \( L_i \) può essere usato
  ai livelli superiori.

\end{definition}
\noindent
Implementare \( L \) equivale a realizzare la macchina astratta \( M_L \). Per realizzare
\( M_L \) ci sono due modi:
\begin{enumerate}
  \item \label{item:compilazione}
    \textbf{Compilazione}: Tradurre \( L \) in un linguaggio macchina sottostante
  \item \label{item:interpretazione}
    \textbf{Interpretazione}: Eseguire \( L \) sulla macchina sottostante
\end{enumerate}

\begin{definition}[Compilazione]
  La compilazione è una traduzione esplicita di un linguaggio \( L \) in un linguaggio
  \( L' \).
\end{definition}

\begin{definition}[Interpretazione]
  L'interpretazione è una traduzione implicita di un linguaggio \( L \) che viene
  \textbf{simulato} in \( L' \).
\end{definition}

\subsection{Notazione}
La notazione che verrà utilizzata per descrivere i programmi è la seguente:
\begin{itemize}
  \item \( Prog^L \): Insieme dei programmi scritti in \( L \)
  \item \( D \): Insieme dei dati manipolati dai programmi in \( L \) 
  \item \( P^L \in Prog^L \quad P^L: D \to D \):
    \[
      \forall input \in D.\; P^L(input) = output \in D
    \] 
    cioè il programma \( P^L \) è una funzione che prende in input un dato e restituisce
    un dato in output.
\end{itemize}

\subsection{Interprete}
L'interprete è la realizzazione della macchina astratta con soluzione interpretativa
(\ref{item:interpretazione}).
\begin{definition}[Interprete]
  Un interprete per il linguaggio \( L \) (scritto nel linguaggio \( L_0 \)) è un programma
  \( I^{L_0L}: (Prog^L \times D) \to D \) che prende in input un programma \( P^L \) e un dato
  e restituisce in output il risultato dell'esecuzione di \( P^L \) su \( input \).
  Questa è la realizzazione della \textbf{macchina di Turing universale}.
  \[
    \underbrace{I^{L_0L}(P^L, input) = output \quad t.c. \quad P^L(input) = output}
  \] 
  \[
    I^{L_0L}(P^L, input) = P^L(input) \Rightarrow \text{Macchina universale}
  \] 
\end{definition}

\subsubsection{Struttura dell'interprete}
La struttura dell'interprete è data dal seguente pseudocodice:
\begin{lstlisting}
begin
  go = true;
  while go do
    fetch(opcode, opinfo) at PC; // Program counter
    decode(opcode, opinfo)

    if opcode needs arguments then
      fetch(args)

    case opcode
      opcode1: execute(opcode1, args)
      opcode2: execute(opcode2, args)
      ...
      halt: go = false

    if opcode has result then store(result)

    PC = PC + sizeof(opcode)

end
\end{lstlisting}

\begin{example}
  Un semplice autointerprete, cioè un interprete scritto nel linguaggio che interpreta.
\begin{lstlisting}[language=Python]
input P, d;
pc = 2;
store = [in -> d, out -> 0, x_1 -> 0, x_2 -> 0, ...];
while pc < length(P) do
  # Find the pc-th instruction
  instruction = lookup(P, pc);

  # Syntax rules
  case instruction of
    skip: pc = pc + 1;
    x = e: store = store[x -> eval(e, store)]; pc = pc + 1;
    ...
  endcase
endwhile;

output store[out]


# Semantic rules
eval(e, store) = case e of
  constant: e
  variable: store(e)
  e1 + e2: eval(e1, store) + eval(e2, store)
  e1 - e2: eval(e1, store) - eval(e2, store)
  e1 * e2: eval(e1, store) * eval(e2, store)
  ...
endcase
\end{lstlisting}
Dove \texttt{in -> d} rappresenta l'assegnamento del dato di input alla variabile \texttt{in}.
\end{example}
\noindent
L'interprete \textbf{realizza} la macchina astratta \( M_L \) di un
linguaggio di programmazione \( L \) partendo da un linguaggio \( L_0 \) 
con una macchina astratta \( M_0 \) direttamente eseguibile passando
attraverso una macchina intermedia \( M_I \) (interprete).

\subsection{Compilatore}
\begin{definition}
  Il compilatore da \( L \) a \( L_0 \) è un programma del tipo:
  \[
    C^{LL_0}: Prog^L \to Prog^{L_0}
  \] 
  che prende in input un programma \( P^L \) e restituisce in output un programma \( P^{L_0} \).
  \[
    \forall P^L \in Prog^L \;.\; C^{LL_0}(P^L) = P^{L_0} \quad t.c. \quad
    \forall d \in D \;.\; P^{L_0}(d) = P^L(d)
  \] 
  Non ci sono accessi alla memoria o a dati. Verificare che due programmi
  siano uguali è un problema non decidibile, quindi il compilatore deve
  essere costruito in modo da garantire che i programmi compilati siano
  equivalenti a quelli originali.
\end{definition}

\end{itemize}

\subsubsection{Prima proiezione di Futamura}
Un compilatore può essere ottenuto \textbf{specializzando} un interprete
su un programma.

\begin{definition}[Specializzatore]
  Uno specializzatore per un linguaggio di programmazione \( L \) è un
  programma che realizza la funzione:
  \[
    Spec_L: (Prog^L \times D) \to Prog^L
  \] 
  tale che:
  \[
    \begin{aligned}
      & \forall P^L \in Prog^L, \forall d \in D \;.\; \\
      &\quad Spec_L(P^L, d) = P^L_d\\
      &\quad \; t.c. \; P^L(d, input) = P^L_d(input)
    \end{aligned}
  \] 
  Quindi \( d \) non è più un dato di input, ma è incorporato nel codice.

  \vspace{1em}
  \noindent
  Lo specializzatore fornisce un legame tra interprete e compilatore. Infatti
  specializzando un interprete (scritto in \( L \)) si ottiene un
  compilatore da \( L \) a \( L_0 \).
  \[
    I(P^L, d) \Rightarrow Spec(I, P^L)
  \] 
  \( Spec \) è un programma scritto in \( L_0 \) che integra nel codice
  l'esecuzione di \( P^L \).
\end{definition}

\section{Descrivere i linguaggi di programmazione}
I linguaggi di programmazione sono dei linguaggi, quindi hanno in caratteristiche
in comune con i linguaggi e quindi anche quelli naturali. Le caratteristiche sono:
\begin{itemize}
  \item \textbf{Sintassi}: Regole di formazione. Determinano quali sono
    le frasi ben formate nel linguaggio e descrive relazioni tra simboli

  \item \textbf{Semantica}: Attribuzione del \textbf{significato} ai simboli.
    Determina il significato delle frasi ben formate nel linguaggio e descrive
    relazioni tra simboli e significati.

  \item \textbf{Pragmatica}: Regole di buon utilizzo del linguaggio.

  \item \textbf{Implementazione} (Caratteristica aggiunta per i linguaggi di programmazione):
    Esistono strumenti che eseguono frasi ben formate del linguaggio.
\end{itemize}

\begin{example}
  Il numero è soltanto sintassi, mentre il significato è il valore che assume:
  \[
    4 \quad \text{(sintassi)} \to \text{significato} \quad 4
  \] 
  \[
    IV \quad \text{(sintassi)} \to  \text{significato} \quad 4
  \] 
\end{example}

\subsection{Sintassi}
Le componenti principali della sintassi sono:
\begin{itemize}
  \item \textbf{Parola}: stringa su un alfabeto \( \Sigma \)

  \item \textbf{Frase}: sequenza di parole ben formata, cioè che rispetta le regole
    della sintassi

  \item \textbf{Linguaggio}: insieme di frasi ben formate

    \begin{example}
      Prendiamo ad esempio il seguente linguaggio:
      \[
        \left\{ a^n b^n \;\left|\; n \in \mathbb{N} \right. \right\}
      \] 
      \begin{itemize}
        \item \textbf{Alfabeto}: \( \Sigma = \left\{ a, b \right\} \) 
        \item \textbf{Parole}: \( \varepsilon, a, b, aa, ab, ba, bb, aaa, aab, ... \)
        \item \textbf{Linguaggio}: \(
          \left\{ a^n b^n \;\left|\; n \in \mathbb{N} \right. \right\}
        \)
      \end{itemize}
    \end{example}

  \item \textbf{Lessema}: una parola (costituita da soli terminali nell'alfabeto) che
  ha un significato, cioè è una parola che rappresenta un concetto o un oggetto nel
  linguaggio.

  \begin{example}
    Alcuni esempi di lessemi sono:
    \begin{itemize}
      \item \texttt{if exp then com else com}:\\
        I lessemi sono \texttt{if}, \texttt{then}, \texttt{else}, mentre \texttt{exp}
        e \texttt{com} non sono lessemi perchè rappresentano un'espressione e un comando,
        cioè sono dei simboli non terminali.
    \end{itemize}
  \end{example}

  \item \textbf{Token}: categorie sintattiche, cioè simboli non terminali

    \begin{example}
      Alcuni esempi di token sono:
      \item \texttt{if exp then com else com}:\\
        I token sono \texttt{exp} e \texttt{com} perchè sono simboli non terminali.
    \end{example}
\end{itemize}

\subsubsection{Strumenti di definizione della sintassi}
Gli strumenti per definire la sintassi di un linguaggio di programmazione sono:
\begin{itemize}
  \item \textbf{Riconoscitore}: è uno strumento per il \textbf{riconoscimento} di
    stringhe appartenenti al linguaggio. Ad esempio gli automi a stati finiti, utilizzati
    nell'analisi lessicale per il riconoscimento dei lessemi.

  \item \textbf{Generatore}: è uno strumento per la \textbf{generazione} di stringhe
    appartenenti al linguaggio. Ad esempio le grammatiche CF, utilizzate nella fase
    di parsing della compilazione per riconoscere programmi scritti in un linguaggio
    di programmazione.
\end{itemize}

\begin{define}
  Un linguaggio di programmazione è dipendente dal contesto, quindi si cerca di svilupparli
  "a livelli", cioè si fa prima tutto ciò che si può fare con gli automi, poi tutto ciò
  che si può fare con le grammatiche CF, e così via.
\end{define}

\begin{definition}
  Una grammatica context free è definita da:
  \[
    G = \left<V, T, P, S \right>
  \] 
  dove:
  \begin{itemize}
    \item \( V \): è un insieme finito di simboli non terminali (variabili)
    \item \( T \): è un insieme finito di simboli terminali
      \[
        V \cap T = \varnothing
      \] 
    \item \( P \): è un insieme finito di procedure:
      \[
        A \to \alpha \quad A \in V, \alpha \in (V \cup T)^*
      \] 
    \item \( S \in V \): è il simbolo iniziale
  \end{itemize}
  In un linguaggio di programmazione la grammatica CF è mappata nel seguente modo:
  \begin{itemize}
    \item \( V \): categorie sintattiche (ad esempio le espressioni, le variabili, ecc...)
    \item \( T \): vocabolario di parole (\texttt{if}, \texttt{then}, \texttt{else}, ecc...)
    \item \( P \): regole di formazione
    \item \( S \in V \): categoria della frase (il programma)
  \end{itemize}

  \vspace{1em}
  \noindent

\end{definition}

\begin{define}[Notazione BNF]
  Per i linguaggi di programmazione viene utilizzata la notazione BNF (Backus-Naur Form)
  che cambia i simboli utilizzati per rappresentare i componenti della grammatica CF,
  ad esempio:
  \[
    \to \quad \text{diventa} \quad ::=
  \] 
  Un linguaggio BNF è CF.
\end{define}

\begin{example}
  Costruiamo una grammatica per espressioni booleane:
  \[
    G = \left< \{Exp\}, \{0, 1, or, and, not, (, )\}, P, Exp \right>
  \] 
  \[
    \begin{aligned}
      P &= \left\{
        \begin{aligned}
        & Exp \to 0 \\
        & Exp \to 1 \\
        & Exp \to (Exp \; or \; Exp) \\
        & Exp \to (Exp \; and \; Exp) \\
        & Exp \to (not \; Exp) \\
        \end{aligned}
      \right.\\
        &= 0 | 1 | (Exp \; or \; Exp) | (Exp \; and \; Exp) | (not \; Exp)
    \end{aligned}
  \] 
\end{example}

\subsubsection{Definizione non formale}
Un linguaggio si può definire anche in un modo non formale, definendo
i costrutti base:
\begin{itemize}
  \item Niente dichiarazioni
  \item Solo variabili ed espressioni booleane
  \item Istruzioni:
    \begin{itemize}
      \item Assegnamento
      \item Composizione sequenziale
      \item Comando iterativo (ciclo)
      \item Comando condizionale (if)
    \end{itemize}
\end{itemize}
Il linguaggio è definito su livelli, partendo dal basso verso l'alto:
\begin{enumerate}
  \item Notazione della grammatica

  \item Linguaggio regolare per riconoscere gli identificatori e le parole
    riservate al linguaggio di programmazione

  \item Espressioni booleane che parte dai simboli definiti ai livelli
    sottostanti
    \[
      \color{blue}B\color{black} \to true \mid false \mid
      (\color{blue}B\color{black} \; or \; \color{blue}B\color{black}) \mid
      (\color{blue}B\color{black} \; and \; \color{blue}B\color{black}) \mid
      (not \; \color{blue}B\color{black}) \mid x
    \] 

  \item Comandi atomici, ad esempio l'assegnamento. Utilizza gli identificatori,
    un simbolo di assegnamento e un espressione (booleana in questo caso).
    \[
      \color{green}A\color{black} \to x := \color{blue}B\color{black}
    \] 

  \item Comandi, ad esempio l'if o il ciclo. Utilizzano i comandi atomici
    e le espressioni booleane.

    \[
      \color{purple}C\color{black} \to \color{green}A\color{black} \mid
      \color{purple}C\color{black} ; \color{purple}C\color{black} \mid
      while \; \color{blue}B\color{black} \; do \; \color{purple}C\color{black} \mid
      if \; \color{blue}B\color{black} \; then \color{purple}C\color{black} \; else
      \; \color{purple}C\color{black}
    \] 

  \item Programmi, cioè sequenze di comandi.
    \[
      \color{orange}P\color{black} \to
      \left\{ \color{purple}C\color{black} \right\}
    \] 
\end{enumerate}

  \centering
  \includegraphics[width=0.8\textwidth]{livelli}
  \caption{Rappresentazione grafica dei livelli di un linguaggio di programmazione}
\end{figure}
Si utilizzano espressioni parentesizzate per evitare ambiguità. Se non
si utilizzassero le parentesi si potrebbe scegliere di eseguire prima
un'operatore invece di un altro e questo porterebbe a risultati diversi
dalla stessa espressione.


\subsection{Analisi semantica}
L'analisi semantica verifica i \textbf{vincoli sintattici}, ad esempio che il numero
di parametri formali sia uguale al numero di parametri attuali, oppure
che l'uso di una variabile sia possibile solo dopo che sia stata dichiarata.
Questi sono vincoli sul codice, quindi \textbf{non catturati dalla grammatica} e si
dicono \textbf{contestuali}.
\begin{itemize}
  \item Sintassi \( \to \) Grammatica CF
  \item Semantica \( \to \) Tutti i processi di verifica ed esclusione che non sono
    descritti dalla sintassi
\end{itemize}

\subsubsection{Induzione strutturale}
L'induzione strutturale permette di dimostrare proprietà di oggetti descritti
in modo induttivo o ricorsivo. Una grammatica descrive in modo induttivo il
linguaggio (insieme di stringhe) che genera, quindi si può utilizzare l'induzione
strutturale per dimostrare proprietà delle stringhe generate.

\begin{definition}
  Per dimostrare una proprietà \( \Pi \) su tutti gli elementi di un insieme
  definito in modo strutturale da una grammatica bisogna:
  \begin{enumerate}
    \item (Caso base): Dimostrare la proprietà su tutti gli elementi base, cioè quelli
      che sono esplicitamente inseriti nell'insieme di stringhe generate dalla
      grammatica.

    \item (Passo induttivo): Dimostrare la proprietà sugli elementi composti sotto \textbf{l'ipotesi
      induttiva} che la proprietà valga per gli elementi che la compongono
  \end{enumerate}
\end{definition}

\vspace{1em}
\noindent

\begin{example}
  Definiamo l'insieme \( X \subseteq \mathbb{N} \):
  \[
  \begin{cases}
    \text{Base} &: 0 \in X \\
    \text{Passo induttivo} &: \text{se } n \in X \text{ allora } n + 3 \in X
  \end{cases}
  \] 
  La proprietà è \( X = 3 \mathbb{N} \) 

  \vspace{1em}
  \noindent
  La proprietà dimostrabile in modo strutturale è: \( X \subseteq 3 \mathbb{N} \):
  \begin{enumerate}
    \item \textbf{Caso base}: \( 0 \) è la base di \( X \) e \( 0 \in 3 \mathbb{N} \) 
    \item \textbf{Passo induttivo}: dimostriamo \( \forall n.\; n \in X \Rightarrow n \in 3 \mathbb{N} \).\\
      Prendiamo \( n \in X \), l'ipotesi induttiva è la seguente:
      \[
        n \in X \Rightarrow n \in 3 \mathbb{N}
      \] 
      bisogna dimostrare che \( n+3 \in 3 \mathbb{N} \). Quindi sappiamo che
      \[
        \begin{aligned}
          n \in  3 \mathbb{N} &\Rightarrow \exists i .\; n = 3i\\
                              &\Rightarrow n + 3 = 3i + 3 = 3(i + 1) \in 3 \mathbb{N}
        \end{aligned}
      \] 
  \end{enumerate}
\end{example}

\begin{example}
  Definiamo induttivamente gli alberi binari in cui i nodi sono numeri \( n \in \mathbb{N} \).
  Ogni numero è un albero binario, perchè è un albero con un solo nodo:
  \[
    n \in BT \quad \text{(Base)}
  \] 
  Consideriamo due alberi binari \( T_1 \in BT \) e \( T_2 \in BT \) allora:
  \begin{figure}[H]
    \centering
    \begin{forest}
      [n
        [\( T_1 \)]
        [\( T_2 \)]
      ]
    \end{forest}
    \( \in BT \) 
    \caption{Albero binario composto da due alberi binari}
  \end{figure}
  Quindi:
  \[
    br(T_1, T_2) \in BT
  \] 
  dove \( br() \) è l'operatore che aggiunge un nodo una radice comune a \( T_1 \) 
  e \( T_2 \) 


  \vspace{1em}
  \noindent
  \textbf{IMPORTANTE:} Dimostriamo che la funzione che calcola il numero di foglie
  (\texttt{\#foglie(T)}) e la funzione che restituisce il numero di nodi che
  hanno figli (\texttt{\#branch(T)}) sono uguali a:
  \[
    \forall T \in BT.\; \texttt{\#foglie}(T) = \texttt{\#branch}(T) + 1
  \] 
  Definiamo la funzione \texttt{\#foglie(T)}:
  \[
    \left\{
    \begin{aligned}
      &\#foglie(n) = 1 \\
      &\#foglie(br(T_1, T_2)) = \#foglie(T_1) + \#foglie(T_2)
    \end{aligned}
    \right.
  \] 
  Definiamo la funzione \texttt{\#branch(T)}:
  \[
    \left\{
    \begin{aligned}
      &\#branch(n) = 0 \\
      &\#branch(br(T_1, T_2)) = \#branch(T_1) + \#branch(T_2) + 1
    \end{aligned}
    \right.
  \]
  Dimostrazione per induzione sulla costruzione degli elementi di \( BT \):
  \begin{enumerate}
    \item \textbf{Caso Base}: è il nodo che non ha figli in BT:
      \[
        T = n \in BT
      \] 
      Sappiamo che:
      \[
        \#foglie(T) = \#foglie(n) = 1 = 1 + 0 = 1 + \#branch(T)
      \] 
      e questo soddisfa la proprietà.

    \item \textbf{Passo induttivo}: \( T = br(T_1, T_2) \). Ipotesi
      induttiva: supponiamo che \( T_1 \) e \( T_2 \) soddisfino la proprietà,
      quindi:
      \[
        \begin{aligned}
          \#foglie(T_1) &= \#branch(T_1) + 1 \\
          \#foglie(T_2) &= \#branch(T_2) + 1
        \end{aligned}
      \] 
      Dobbiamo dimostrare che \( \#foglie(T) = \#branch(T) + 1 \):
      \[
        \begin{aligned}
          \#foglie(T) &\stackrel{def\; T}{=} \#foglie(br(T_1, T_2)) \\
                      &\stackrel{def \; \#foglie}{=} \#foglie(T_1) + \#foglie(T_2) \\
                      &\stackrel{Hp\; Ind}{=}
                  \underbrace{\#branch(T_1) + 1 + 
                  \#branch(T_2)}_{def\; \#branch(br(T_1, T_2))}
                  + 1 \\
                      &= \#branch(\underbrace{br(T_1, T_2)}_{T}) + 1 \\
                      &= \#branch(T) + 1
        \end{aligned}
      \] 
      Quindi anche \( T \) soddisfa la proprietà.
  \end{enumerate}
\end{example}

\begin{example}
  Consideriamo la seguente grammatica:
  \[
    S \to aa \mid bb \mid aSa \mid bSb
  \] 
  questa grammatica genera tutte le stringhe palindrome di lunghezza pari.
  Siccome \( aa \) e \( bb \) sono nel linguaggio, se si prende una qualsiasi
  stringa nel linguaggio \( S \) e si aggiungono \( a \) o \( b \) all'inizio e
  alla fine, si ottiene una stringa ancora palindroma, quindi anche questa stringa è
  nel linguaggio. Si può quindi definire questa grammatica in modo induttivo:
  \[
    LS: 
    \begin{cases}
      aa, bb, \in LS \quad \text{(Base)} \\
      \sigma \in LS \Rightarrow a\sigma a, b\sigma b \in LS \quad \text{(Passo induttivo)}
    \end{cases}
  \] 
  Quindi \( L(S) = LS \) e la proprietà da dimostrare è:
  \[
    \sigma \in LS \Rightarrow \exists x, y.\; \sigma = x y \land x = reverse(y)
  \] 

  \begin{enumerate}
    \item \textbf{Caso base}: Prendiamo:
      \[
        aa = \sigma \Rightarrow x = a, y = a \quad \text{e} \quad
        x = a = reverse(a) = reverse(y)
      \] 
      analogo per \( \sigma = bb \).

    \item \textbf{Passo induttivo}: Prendiamo \( \sigma \in LS \), l'ipotesi
      induttiva è la seguente:
      \[
        \sigma = xy \;t.c.\; x = reverse(y)
      \] 
      Prendiamo \( \sigma' = a \sigma a \). Per ipotesi induttiva:
      \[
        \sigma' = a \sigma a = a \overbrace{xy}^{\sigma} a = x'y'
      \] 
      con \( x' = ax \) e \( y' = ya \). Quindi:
      \[
        \begin{aligned}
          x' = ax &= a reverse(y) \\
                  &= reverse(\underbrace{ya}_{y'}) \\
                  &= reverse(y')
        \end{aligned}
      \] 
      Abbiamo dimostrato che \( \sigma' \) è palindroma, quindi \( \sigma' \in LS \).
  \end{enumerate}
\end{example}

\begin{exercise}
  Consideriamo la seguente grammatica:
  \[
    E \to n \mid E + E \mid E * E
  \] 
  \begin{enumerate}
    \item Definire strutturalmente l'insieme \( Exp \)

    \item Si considerino le seguenti funzioni:
      \begin{itemize}
        \item Conta il numero di operatori:\\
          \( op(E): op(n) = 0 \quad op(E_1 + E_2) = op(E_1 * E_2) = 1 + op(E_1) +
          op(e_2) \)

        \item Conta il numero di valori:\\
          \( val(E): val(n) = 1 \quad val(E_1 + E_2) = val(E_1 * E_2) = val(E_1) + val(E_2) \)
      \end{itemize}
      Dimostrare che:
      \[
        \forall E \;.\; val(E) = op(E) + 1
      \] 
  \end{enumerate}

  Definizione strutturale dell'insieme \( Exp \):
  \begin{itemize}
    \item \textbf{Base}: \( n \in L(E) \)
    \item \textbf{Passo induttivo}: se \( e_1, e_2 \in Exp \) allora
      \( e_1 + e_2, e_1 * e_2 \in Exp \).
      Le espressioni sono il minimo insieme chiuso che soddisfa queste condizioni,
      quindi non ci sono altre espressioni in \( L(E) \). Quindi ad esempio:
      \[
        3 + 5 * 2 \in Exp
      \] 
  \end{itemize}
  Sappiamo che:
  \[
    \begin{aligned}
      &\begin{cases}
        val(n) = 1 \\
        val(e_1 + e_2) = val(e_1 * e_2) = val(e_1) + val(e_2)
      \end{cases} \\
      &\begin{cases}
        op(n) = 0 \\
        op(e_1 + e_2) = op(e_1 * e_2) = op(e_1) + op(e_2) + 1
      \end{cases}
    \end{aligned}
  \] 
  Quindi sull'esempio precedente:
  \[
    \begin{aligned}
      val(\underbrace{3 + 5}_{e_1} * \underbrace{2}_{e_2}) &= val(3 + 5) + val(2) \\
                                                           &= val(3) + val(5) + val(2) \\
                                                           &= 1 + 1 + 1 = 3
    \end{aligned}
  \] 
  \[
    \begin{aligned}
      op(\underbrace{3 + 5}_{e_1} * \underbrace{2}_{e_2}) &= op(3 + 5) + op(2) + 1 \\
                                                          &= op(3) + op(5) + op(2) + 1 + 1 \\
                                                          &= 0 + 0 + 0 + 1 + 1 = 2
    \end{aligned}
  \] 
  Dimostriamo che \( \Pi:\; \forall e \in Exp \;.\; val(e) = op(e) + 1 \) per induzione
  strutturale su \( Exp \) 
  \begin{itemize}
    \item \textbf{Caso base}: Verifichiamo \( \Pi  \) sugli elementi esplicitamente
      inseriti nell'insieme \( Exp \), cioè sui numeri \( n \in \mathbb{N}\):
      consideriamo le espressioni \( e = n \) 
      \[
        val(e) = val(n) = 1 \quad \textbf{=} \quad 1 + 0 = 1 + op(n) = 1 + op(e)
      \] 

    \item \textbf{Ipotesi induttiva}: Dimostriamo per gli elementi composti supponendo
      l'ipotesi induttiva sugli elementi che li compongono. Sia \( e = e_1 + e_2 \) (ipotesi1)
      (analogo per \( e = e_1 * e_2 \)), l'ipotesi induttiva è la seguente:
      \[
        val(e_1) = op(e_1) + 1 \quad \land \quad val(e_2) = op(e_2) + 1
      \] 
      Dobbiamo dimostrare che \( val(e) = op(e) + 1 \):
      \[
        \begin{aligned}
          val(e) &\stackrel{ipotesi1}{=} val(e_1 + e_2) \\
                 &\stackrel{def\; val}{=} val(e_1) + val(e_2) \\
                 &\stackrel{Hp.\; Ind.}{=} \underbrace{op(e_1) + 1 + op(e_2)}_{op(e_1 + e_2)} + 1 \\
                 &\stackrel{def\; op}{=} op(e_1 + e_2) + 1 \\
                 &\stackrel{ipotesi1}{=} op(e) + 1
        \end{aligned}
      \] 
  \end{itemize}
  Abbiamo dimostrato che \( \Pi \) è vera per tutti gli elementi di \( Exp \).
\end{exercise}

\subsubsection{Semantica}
La semantica è l'attribuzione di significato ai simboli.
Il significato è un'entità autonoma che esiste indipendentemente dai simboli
usati per descriverlo. Esistono 3 tipi di semantica:
\begin{itemize}
  \item \textbf{Denotazionale}: È un modello completamente matematico che descrive il
    programma come una funzione su stati (programmi come funzioni). Lo \textbf{stato}
    descrive il valore di tutte le variabili in un certo momento.
    \[
      Stato: Var \mapsto Val
    \] 
    La semantica denotazionale è una funzione che prende un programma e restituisce
    una funzione che prende uno stato iniziale e restituisce uno stato finale:
    \[
      E: Prog \mapsto \underbrace{(Stato \to Stato)}_{\substack{\text{Denotazione associata}\\\text{al programma}}}
    \] 
    Un esempio di semantica denotazionale è il comportamento input/output

  \item \textbf{Operazionale}: Descrive il significato del programma come sequenza
    di trasformazione di stati, \textbf{eseguendo} i suoi comandi sulla macchina astratta.
    Per semplificare vedremo lo stato come se fosse la memoria:
    \[
      \text{Memoria: } \sigma: Var^n \mapsto Val^n
    \] 
    \[
      \text{Stato: } (Prog \times Mem) \cup Mem
    \] 
    dove:
    \begin{itemize}
      \item \( Prog \) è il programma che stiamo eseguendo
      \item \( Mem \) è la memoria su cui stiamo eseguendo il programma
      \item \( \cup Mem \) è l'insieme degli stati finali
    \end{itemize}
    Ad esempio:
    \[
      \text{Variabili:} <x, y, z> \mapsto <2, 5, 3>
      \leadsto <x \mapsto 2, y \mapsto 5, z \mapsto 3>
    \] 
    Per eseguire un programma si utilizza un \textbf{sistema di transizione}:
    \[
      <\Sigma, \to>
    \] 
    dove:
    \begin{itemize}
      \item \( \Sigma \) è l'insieme degli stati
      \item \( \to \) è la relazione di transizione che descrive come si passa da
        uno stato ad un altro eseguendo un comando del programma
    \end{itemize}

  \item \textbf{Assiomatica}: Descrive il programma attraverso le proprietà che
    soddisfa. Si calcola utilizzando un sistema di derivazione che fornisce regole per
    dimostrare proprietà sui programmi:
    \begin{figure}[H]
      \centering
      \includegraphics[width=0.6\textwidth]{assiomatica}
      \caption{Esempio di sistema di derivazione per la semantica assiomatica}
    \end{figure}
    \noindent
    Quindi descrive relazioni tra proprietà di stati.
    Ad esempio fornendo le proprietà invarianti dei programmi, cioè le
    proprietà che sono vere prima e dopo l'esecuzione di un programma.
\end{itemize}

\begin{example}
  Consideriamo il seguente programma:
  \[
  P = \begin{cases}
    z := 2; \\
    y := z; \\
    y := y + 1; \\
    z := y;
  \end{cases}
  \] 
  Applichiamo i tre tipi diversi di semantica al programma \( P \):
  \begin{itemize}
    \item \textbf{Semantica denotazionale}:
      \[
        \begin{aligned}
          &\mathcal{D}[\overbrace{P}^{Prog}]
          (\overbrace{z \mapsto \bot, y \mapsto \bot}^{Stato\; iniziale}) = \\
          &= \mathcal{D}[z := 2; y := z; y := y + 1; z := y](z \mapsto \bot, y \mapsto \bot) \\
          &= \mathcal{D}[z := y] \circ \mathcal{D}[y := y + 1] \circ \mathcal{D}[y := z] \circ
          \color{blue}\mathcal{D}[z := 2] (z \mapsto \bot, y \mapsto \bot) \\
          &= \mathcal{D}[z := y] \circ \mathcal{D}[y := y + 1] \circ
          \color{blue}\mathcal{D}[y := z] (z \mapsto 2, y \mapsto \bot) \\
          &= \mathcal{D}[z := y] \circ
          \color{blue}\mathcal{D}[y := y + 1] (z \mapsto 2, y \mapsto 2) \\
          &= \color{blue}\mathcal{D}[z := y] (z \mapsto 2, y \mapsto 3) \\
          &= \color{blue}(z \mapsto 3, y \mapsto 3)
        \end{aligned}
      \] 
      Quindi \( \mathcal{D}[P](z \mapsto \bot, y \mapsto \bot) = (z \mapsto 3, y \mapsto 3) \)
      è usata per:
      \begin{itemize}
        \item Dimostrare proprietà (correttezza)
        \item Fornisce uno strumento formale per ragionare sulle funzionalità dei
          programmi
        \item Non è utile a chi usa il linguaggio di programmazione
      \end{itemize}

    \item \textbf{Semantica assiomatica}: Fornisce proprietà relazionali sull'input
      e sull'output del programma, ad esempio:
      \begin{figure}[H]
        \centering
        \includegraphics[width=0.6\textwidth]{esempio_assiomatica}
        \caption{Esempio di proprietà assiomatica per il programma \( P \)}
      \end{figure}

    \item \textbf{Semantica operazionale}:
      \[
        \begin{aligned}
          &<P, (z \mapsto \bot, y \mapsto \bot)> = \\
          &= <\color{blue}z := 2\color{black}; y := z; y := y + 1; z := y, (z \mapsto \bot, y \mapsto \bot)> \\
          &= <\color{blue}y := z\color{black}; y := y + 1; z := y, (z \mapsto 2, y \mapsto \bot)> \\
          &= <\color{blue}y := y + 1\color{black}; z := y, (z \mapsto 2, y \mapsto 2)> \\
          &= <\color{blue}z := y\color{black}, (z \mapsto 2, y \mapsto 3)> \\
          &= <z \mapsto 3, y \mapsto 3>
        \end{aligned}
      \] 
  \end{itemize}
\end{example}

\subsubsection{Composizionalità della semantica}
Il significato di ogni programma è funzione del significato dei suoi componenti
immediati, cioè si possono ottenere le proprietà semantiche di un programma a partire
dalle proprietà semantiche dei suoi componenti.

\subsubsection{Equivalenza di programmi}
La semantica viene utilizzata per verificare l'equivalenza tra programmi.
\begin{itemize}
  \item \( P_1 = P_2 \) dice che \( P_1 \) è esattamente lo stesso programma di \( P_2 \),
    cioè sono sintatticamente uguali. Questa si dice \textbf{uguaglianza sintattica}.

  \item \( P_1 \equiv P_2 \) dice che \( P_1 \) e \( P_2 \) hanno lo stesso comportamento
    (stesso significato), anche se sono scritti in modo diverso. Questa si dice
    \textbf{equivalenza} o uguaglianza semantica.
\end{itemize}

\begin{example}
  Consideriamo i seguenti programmi:
  \begin{itemize}
    \item \( P_1 \) insertion sort
    \item \( P_2 \) bubblue sort
  \end{itemize}
  Sappiamo che i due programmi sono diversi, quindi:
  \[
    P_1 \neq P_2
  \] 
  Ma sappiamo che entrambi ordinano una lista, quindi il loro comportamento è lo stesso:
  \begin{itemize}
    \item Con la semantica denotazionale si ha:
      \[
        \mathcal{D}[P_1] = \mathcal{D}[P_2] \leadsto P_1 \equiv P_2
      \] 

    \item Con la semantica operazionale si ha:
      \[
        [\![ P_1 ]\!] \neq [\![ P_2 ]\!] \leadsto P_1 \not\equiv P_2
      \] 
  \end{itemize}
  Quindi l'equivalenza dipende dalla scelta della semantica
\end{example}

\subsubsection{Categorie sintattiche}
Le categorie sintattiche sono insiemi di stringhe che condividono una certa proprietà
per:
\begin{itemize}
  \item Manipolare dati (espressioni)
  \item Trasformare stati (comandi)
  \item Creare e manipolare i legami (dichiarazioni), ad esempio variabile-valore,
    variabile-tipo, funzione-codice, ecc...
\end{itemize}
Si hanno quindi:
\begin{itemize}
  \item \textbf{Stato}: descrive gli effetti sugli identificatori
  \item \textbf{Ambiente}: è un insieme di legami tra identificatori e oggetti
    denotati
\end{itemize}

\vspace{1em}
\noindent
Le categorie sintattiche principali sono:
\begin{itemize}
  \item \textbf{Espressioni}: denotano e manipolano valori. Le espressioni possono
    essere complesse e quindi devono essere \textbf{valutate} per restituire il
    singolo valore che sta denotando.
    Espressioni \textbf{sintatticamente diverse} possono denotare lo stesso valore.
    Ad esempio \( 1 + 5 \) e \( 3 * 2 \) denotano lo stesso valore.

    \begin{definition}
      Due espressioni sono equivalenti se vengono valutate nello \textbf{stesso}
      valore a partire da qualunque stato/memoria iniziale.
    \end{definition}

  \item \textbf{Comandi}: denotano richieste di modifica della memoria.
    \begin{itemize}
      \item Le strutture di controllo decidono quali comandi eseguire
      \item Gli assegnamenti modificano effettivamente la memoria
    \end{itemize}
    Rappresentano funzioni: \( Memoria \mapsto Memoria \). I comandi vengono
    \textbf{eseguiti} per attuare la corrispondente richiesta di trasformazione della
    memoria. Queste trasformazioni sono \textbf{irreversibili}, cioè non possono essere
    cancellate, ma solo annullate semanticamente, ad esempio:
    \[
      x := x + 1; \; x := x - 1
    \] 
    Questi comandi si annullano, ma lo stato è stato comunque modificato.

    \begin{definition}
      Due comandi sono equivalenti se per ogni stato iniziale in input producono lo
      stesso stato in output. Quindi l'equivalenza è vista come uguaglianza della
      funzione calcolata.
    \end{definition}

  \item \textbf{Dichiarazioni}: denotano richieste di creazione o modifica di legami,
    ovvero di \textbf{ambienti}. Le dichiarazioni vengono \textbf{elaborate} per
    attuare le richieste di creazione o modifica degli ambienti.

    \begin{definition}
      Due dichiarazioni sono equivalenti se producono lo stesso ambiente (e la stessa
      memoria) a partire da qualunque stato iniziale.
    \end{definition}
\end{itemize}
La separazione tra le categorie sintattiche non è rigida, ad esempio un'espressione
può avere anche side effects, quindi modificare la memoria, ecc...

\subsubsection{Linguaggio "giocattolo" PLO}
In questo corso considereremo un linguaggio di programmazione "giocattolo" 
simile al linguaggio Pascal in cui:
\begin{itemize}
  \item Esiste un singolo tipo di dato \texttt{integer}
  \item I booleani sono rappresentati come interi
  \item Strutture di controllo standard
  \item Astrazione del controllo
  \item Le procedure non hanno parametri
\end{itemize}
Il linguaggio è definito nel seguente modo:
\[
  \begin{aligned}
    \texttt{\color{orange}<Program>} \quad &\texttt{\color{orange}P\color{black}} \to
    \texttt{\color{purple}B} \\
    %
    \texttt{\color{purple}<Block>} \quad &\texttt{\color{purple}B\color{black}} \to
    \texttt{\color{Green}D\color{blue}S} \\
    %
    \texttt{\color{Green}<Declar>} \quad & \texttt{\color{Green}D\color{black}} \to
    \texttt{[const I = N\{,I=N\};]} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{[var I\{,I\};]} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{[procedure I;\color{purple}B\color{black};]} \\
    %
    \texttt{\color{blue}<Stmts>} \quad & \texttt{\color{blue}S\color{black}} \to
    \varepsilon 
    \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{I := \color{gray}E} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{call I} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{if \color{olive}C\color{black}\ then \color{blue}S\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{while \color{olive}C\color{black}\ do \color{blue}S\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{begin \color{blue}S\color{black}\{;\color{blue}S\color{black}\} end} \\
    %
    \texttt{\color{olive}<Cond>} \quad & \texttt{\color{olive}C\color{black}} \to
    \texttt{odd \color{gray}E\color{black}} 
                    \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ > \color{gray}E\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ >= \color{gray}E\color{black}} 
                    \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ = \color{gray}E\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ \# \color{gray}E\color{black}} 
                    \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ < \color{gray}E\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ <= \color{gray}E\color{black}} \\
    %
    \texttt{\color{gray}<Expr>} \quad & \texttt{\color{gray}E\color{black}} \to
    \texttt{I} \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{N} 
                      \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ op \color{gray}E\color{black}} 
                      \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{(\color{gray}E\color{black})}
  \end{aligned}
\] 
\begin{itemize}
  \item I programmi \( \color{orange}P \) sono blocchi \( \color{purple}B \).
  \item I blocchi \( \color{purple}B \) sono una dichiarazione \( \color{Green}D \)
    seguita da un comando \( \color{blue}S \).
    Si noti che \( \color{Green}D \) ed \( \color{blue}S \) sono definiti in modo
    libero dal contesto, ovvero quello che possiamo generare da \( \color{blue}S \)
    non deve avere necessariamente nessun vincolo con quello che generiamo da
    \( \color{Green}D \). In altre parole questa grammatica non ci obbliga in nessun
    modo ad usare identificatori precedentemente definiti. Quindi dalla descrizione
    sintattica rimangono fuori gli aspetti context sensitive (vincoli sintattici)
    che la grammatica CF non è in grado di rappresentare. Servirà la definizione di
    una “semantica statica” che avrà il ruolo di verificare questi vincoli prima di
    compilare e/o eseguire.

  \item Le dichiarazioni \( \color{Green}D \) sono dichiarazioni di valori costanti
    con nome (\texttt{const}), dichiarazioni di variabili (\texttt{var}) e
    dichiarazioni di procedure (ovvero di un blocco riferibile con un nome).

  \item I nomi sono gli identificatori \( I \), considerati qui parte del vocabolario;
    \( N \) sono i numeri naturali, considerati anch’essi parte del vocabolario.

  \item Non abbiamo bisogno del concetto di tipo in quanto abbiamo solo un tipo di dato.

  \item Si noti che le produzioni di \( \color{Green}D \) sono tutte opzionali, questo
    significa che un programma può essere un blocco senza dichiarazioni.

  \item I comandi sono: il comando vuoto, l’assegnamento di un valore (denotato da una
    espressione) ad un identificatore, la chiamata ad una procedura mediante il suo
    nome, il comando condizionale che esegue un comando al verificarsi di una
    condizione booleana \( \color{olive}C \), il ciclo che ripete un comando finché una data
    condizione \( \color{olive}C \) è vera e la composizione sequenziale di comandi.

  \item Le condizioni \( \color{olive}C \) sono espressioni dal valore booleano:
    \texttt{odd} è un predicato unario che verifica se l’espressione è
    0, poi ci sono vari operatori di confronto di espressioni \( \color{gray}E \).

  \item Le espressioni \( \color{gray}E \) sono identificatori, valori naturali,
    operazioni tra espressioni e espressioni tra parentesi.

  \item Si sintetizza con \( \texttt{op} \) la metavariabile dell’insieme
    \( \{+, -, *, /\} \). La stessa cosa si potrebbe fare per gli operatori
    di confronto, mettendo in \( \color{olive}C \) la produzione \( E \; cop \; E \) con
    \( cop \) metavariabile per gli elementi in \( \{ >, >=, =, \#, <=, < \} \).
\end{itemize}
La grammatica delle espressioni è ambigua, ma più comoda da usare e si può utilizzare
per definire la semantica. Per il compilatore però bisogna fornire una grammatica
equivalente precisa e quindi non ambigua, come la seguente:
\[
  \begin{aligned}
    \texttt{\color{gray}<Expr>} \quad &\texttt{\color{gray}E\color{black}} \to
    \texttt{[+,-]\color{gray}
    T\color{black} [\color{gray}A\color{black} \color{gray}T\color{black}]} \\
    %
    \texttt{\color{gray}<Add op>} \quad &\texttt{\color{gray}A\color{black}} \to
    \texttt{+} \color{BrickRed}\mid\color{black} \texttt{-} \\
    %
    \texttt{\color{gray}<Terms>} \quad &\texttt{\color{gray}T\color{black}} \to
    \texttt{\color{gray}F\color{black} [\color{gray}M\color{black}
    \color{gray}F\color{black}]} \\
                                       &\texttt{\color{gray}M\color{black}} \to
                                       \texttt{*} \color{BrickRed}\mid\color{black}
                                       \texttt{/} \\
    %
    \texttt{\color{gray}<Factor>} \quad &\texttt{\color{gray}F\color{black}} \to 
    \color{MidnightBlue}\texttt{I}\color{black} \color{BrickRed}\mid\color{black}
    \color{OliveGreen}\texttt{N}\color{black} \color{BrickRed}\mid\color{black} \texttt{\color{black}(\color{gray}E\color{black})} \\
    %
    \texttt{\color{OliveGreen}{<Numbers>}} \quad &\texttt{\color{OliveGreen}N\color{black}}
    \to \texttt{[+,-]\color{Orchid}d\color{black}\{\color{Orchid}d\color{black}\}} \\
    %
    \texttt{\color{Orchid}{<Digit>}} \quad &\texttt{\color{Orchid}d\color{black}} \to
    \texttt{0} \color{BrickRed}\mid\color{black} \ldots \color{BrickRed}\mid\color{black} \texttt{9} \\
    %
    \texttt{\color{MidnightBlue}{<Ident>}} \quad &\texttt{\color{MidnightBlue}I\color{black}} \to
    \texttt{\color{Maroon}l\color{black}\{\color{Maroon}l\color{black},
    \color{Orchid}d\color{black}\}} \\
    %
    \texttt{\color{Maroon}{<Letter>}} \quad &\texttt{\color{Maroon}l\color{black}} \to
    \texttt{A} \color{BrickRed}\mid\color{black} \ldots \color{BrickRed}\mid\color{black} \texttt{Z}
  \end{aligned}
\]
