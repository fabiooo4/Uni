\section{Linguaggi di programmazione}
Un linguaggio interpretato è un linguaggio di programmazione in cui le istruzioni vengono
eseguite direttamente, senza la necessità di essere compilate in un formato eseguibile.
Un interprete è rappresentato da un modello di esecuzione, cioè un \textbf{control
flow graph} (CFG) che rappresenta il flusso di controllo del programma.

I programmi si possono interpretare in diversi modi:
\begin{itemize}
  \item \textbf{Interpretazione diretta}: l'interprete esegue direttamente le istruzioni del
    programma e segue il flusso di controllo rappresentato dal CFG restituendo i risultati.

  \item \textbf{Interpretazione del segno}: l'interprete esegue le istruzioni del programma
    ma invece di restituire i risultati, restituisce un segno che rappresenta il risultato.
    Questo è utile quando non si riesce a restituire un risultato concreto, ad esempio quando
    un algoritmo non termina.

  \item \textbf{Analisi statica}: l'interprete analizza il programma senza eseguirlo, ad
    esempio per verificare la correttezza del codice o per ottimizzare le prestazioni.
\end{itemize}


\subsection{Obiettivo dei linguaggi di programmazione}
\noindent
L'obiettivo è creare un modello formale che riesca ad eseguire i costrutti necessari
per eseguire un programma. I componenti principali sono:
\begin{itemize}
  \item \textbf{Dati}
  \item \textbf{Algoritmi}: Calcolo da eseguire sui dati, cioè calcola funzioni.
\end{itemize}

\vspace{1em}
\noindent

\begin{definition}[Algoritmo]
    Un algoritmo è una sequenza di passi primitivi finita. L'algoritmo è descritto
    attraverso un programma in un linguaggio di programmazione.
\end{definition}

\vspace{1em}
\noindent

\begin{definition}[Linguaggio di programmazione]
  Un linguaggio di programmazione è uno strumento formale per descrivere algoritmi
  applicati ai dati.
\end{definition}

\vspace{1em}
\noindent

\begin{definition}[Programma]
  Il programma è la realizzazione nel linguaggio di programmazione di un algoritmo
  (concetto astratto).
  È una rappresentazione sintattica, cioè rispetta delle regole sintattiche del
  linguaggio di programmazione, finita di comportamenti potenzialmente infiniti.
  I comportamenti potenzialmente infiniti sono funzioni calcolate dall'algoritmo
  rappresentato (o implementato) dal programma:
  \[
    f: Dom \to Codom \cup \{\uparrow\}
  \] 

  \noindent
  Il linguaggio di programmazione deve considerare strumenti per descrivere
  \textbf{metodi di calcolo}
\end{definition}

\noindent
Alcuni esempi di linguaggi sono:
\begin{itemize}
  \item \textbf{Linguaggio matematico}: notazione rigorosa per rappresentare funzioni.
    Questo linguaggio non è adatto per descrivere algoritmi perchè non ha strumenti
    che descrivono \textbf{come} calcolare funzioni.

    \begin{itemize}
      \item Paradigma funzionale: nascono dal linguaggio matematico per adattarlo
        alla descrizione di algoritmi. Derivano dal lambda calcolo e alcune implementazioni
        sono Haskell o Lisp.
    \end{itemize}


  \item \textbf{Linguaggi logici}: notazione rigorosa con regole-assiomi che permettono
    di specificare un processo di calcolo attraverso regole di derivazione. Il
    calcolo è rappresentato da un processo di dimostrazione. Manca la descrizione
    di calcoli potenzialmente infiniti.

    \begin{itemize}
      \item Paradigma logico: nascono dal linguaggio logico per adattarlo alla descrizione di
        algoritmi. Derivano dal calcolo dei predicati e un esempio di implementazione è
        Prolog.
    \end{itemize}

  \item \textbf{Linguaggi di programmazione}: notazione rigorosa, tramite grammatiche CF,
    con:
    \begin{itemize}
      \item \textbf{Regole}: descrivono la semantica delle istruzioni
      \item \textbf{Istruzioni}: permettono di descrivere i passi di calcolo
    \end{itemize}
\end{itemize}

\begin{example}
  Alcuni esempi di applicazioni dei linguaggi di programmazione sono:
  \begin{itemize}
    \item Applicazioni scientifiche: Fortran, ecc...
    \item Applicazioni economiche: Cobol, ecc...
    \item Applicazioni AI: Lisp, ecc...
    \item Applicazioni sistemi: C, ecc...
    \item Applicazioni web: JavaScript, Java, ecc...
  \end{itemize}
\end{example}

\subsection{Aspetti di progettazione}
Gli aspetti per scegliere un linguaggio di programmazione sono:
\begin{itemize}
  \item \textbf{Readability} (Leggibilità): quanto è facile leggere e comprendere i
    programmi, cioè:
    \begin{itemize}
      \item Sintassi chiara
      \item Nessuna ambiguità
      \item Facilità di analisi del codice
    \end{itemize}

  \item \textbf{Writability} (Facilità di scrittura): quanto è facile o agevole scrivere
    programmi.

  \item \textbf{Reliability} (Affidabilità): quanto il linguaggio è conforme alle specifiche
    e quanto riduce l'introduzione di errori di programmazione.

  \item \textbf{Cost} (Costo):
    \begin{itemize}
      \item Costi di efficienza
      \item Costi di apprendimento
      \item Costi di utilizzo in modo corretto
    \end{itemize}
\end{itemize}

\subsubsection{Tipi di linguaggi}
I linguaggi di programmazione si possono classificare come:
\begin{itemize}
  \item \textbf{Linguaggi di basso livello}: sono linguaggi vicini alla macchina fisica,
    come ad esempio l'assembly o il linguaggio macchina.

  \item \textbf{Linguaggi di alto livello}: sono linguaggi più vicini al linguaggio umano,
    come ad esempio:
    \begin{itemize}
      \item Linguaggi funzionali
      \item Linguaggi imperativi
      \item Linguaggi logici
    \end{itemize}

    \vspace{1em}
    \noindent
    I linguaggi si possono ulteriormente dividere in:
    \begin{itemize}
      \item \textbf{Linguaggi puri}: 
        I linguaggi imperativi e alcuni linguaggi funzionali sono detti \textbf{funzionali impuri}
        perchè permettono di descrivere algoritmi con variabili e assegnamenti che
        possono ricevere degli "effetti collaterali" (side effects) cioè modifiche
        dello stato del programma. Quindi la variabile è un'astrazione della cella di memoria,
        di conseguenza lo stesso nome in punti diversi può avere valori diversi.

      \item \textbf{Linguaggi puri}:
        I linguaggi logici e alcuni linguaggi funzionali sono detti \textbf{funzionali puri}
        perchè le variabili sono matematiche e quindi \textbf{immutabili}, cioè non possono
        essere modificate dopo essere state assegnate. In questo modo si evitano gli
        effetti collaterali.
    \end{itemize}

    \vspace{1em}
    \noindent
    I linguaggi di alto livello si possono dividere in base al paradigma di programmazione:
    \begin{itemize}
      \item \textbf{Object Oriented}: linguaggi che permettono di strutturare il codice
        in oggetti con attributi e metodi.

      \item \textbf{Procedurali}: linguaggi che permettono di strutturare il codice in procedure
        o funzioni.

      \item \textbf{Strutturati}: linguaggi che permettono di strutturare il codice in blocchi di istruzioni
        con costrutti di controllo come if, while, for, ecc...

      \item \textbf{Dichiarativi}: linguaggi che si basano sulla descrizione del problema
        piuttosto che sulla descrizione del processo di calcolo.
    \end{itemize}
\end{itemize}

\subsection{Implementare un linguaggio di programmazione}
Le caratteristiche che hanno effetto sulla progettazione di un linguaggio di programmazione
sono:
\begin{itemize}
  \item L'architettura della macchina su cui è implementato
  \item Le metodologie di progettazione di programmi, cioè i paradigmi di programmazione
\end{itemize}

\section{Esecuzione dei linguaggi}
L'obiettivo dei linguaggi è \textbf{eseguire i programmi}.
Per poter eseguire un programma serve un altro programma che lo esegue che è detto
\textbf{macchina astratta} per il linguaggio di programmazione.

\begin{definition}[Macchina astratta]
  Dato un linguaggio di programmazione \( L \), la macchina astratta per \( L \)
  \( M_L \) è un insieme di \textbf{strutture dati} e \textbf{algoritmi} che permettono
  di memoriazzare e eseguire programmi scritti in \( L \).

  \vspace{1em}
  \noindent
  Intuitivamente è la macchina il cui linguaggio è esattamente \( L \).

  \begin{itemize}
    \item Per ogni macchina astratta \( M_L \) esiste un unico linguaggio \( L \) 
      che è il linguaggio riconosciuto ed eseguito da \( M_L \) 
    \item Per ogni linguaggio di programmazione \( L \) possono esistere
      infinite macchine astratte per \( L \) 
  \end{itemize}
\end{definition}

\begin{example}
  La macchina fisica è la \textbf{macchina astratta} per il linguaggio macchina.
  Mentre tutte le macchine astratte per qualsiasi altro linguaggio di programmazione
  di alto livello
  sono software, quindi altri programmi che traducono le istruzioni del linguaggio di programmazione
  in un'istruzione in un livello più basso.
\end{example}

\begin{definition}[Linguaggio macchina]
  Data una macchina \( M \), il linguaggio macchina di \( M \) \( L_M \) è il linguaggio
  che ha come stringhe quelle direttamente interpretabili da \( M \).

  \vspace{1em}
  \noindent
  Se si ha una pila di macchine astratte \( \{M_{i-1}, M_i, M_{i+1}, M_{i+2}\} \)
  la macchina \( M_i \) ha il suo linguaggio macchina \( L_{i} \) che utilizza il
  linguaggio macchina \( L_{i-1} \) della macchina \( M_{i-1} \) per eseguire i
  programmi scritti in \( L_i \). Inoltre il linguaggio \( L_i \) può essere usato
  ai livelli superiori.

\end{definition}
\noindent
Implementare \( L \) equivale a realizzare la macchina astratta \( M_L \). Per realizzare
\( M_L \) ci sono due modi:
\begin{enumerate}
  \item \label{item:compilazione}
    \textbf{Compilazione}: Tradurre \( L \) in un linguaggio macchina sottostante
  \item \label{item:interpretazione}
    \textbf{Interpretazione}: Eseguire \( L \) sulla macchina sottostante
\end{enumerate}

\begin{definition}[Compilazione]
  La compilazione è una traduzione esplicita di un linguaggio \( L \) in un linguaggio
  \( L' \).
\end{definition}

\begin{definition}[Interpretazione]
  L'interpretazione è una traduzione implicita di un linguaggio \( L \) che viene
  \textbf{simulato} in \( L' \).
\end{definition}

\subsection{Notazione}
La notazione che verrà utilizzata per descrivere i programmi è la seguente:
\begin{itemize}
  \item \( Prog^L \): Insieme dei programmi scritti in \( L \)
  \item \( D \): Insieme dei dati manipolati dai programmi in \( L \) 
  \item \( P^L \in Prog^L \quad P^L: D \to D \):
    \[
      \forall input \in D.\; P^L(input) = output \in D
    \] 
    cioè il programma \( P^L \) è una funzione che prende in input un dato e restituisce
    un dato in output.
\end{itemize}

\subsection{Interprete}
L'interprete è la realizzazione della macchina astratta con soluzione interpretativa
(\ref{item:interpretazione}).
\begin{definition}[Interprete]
  Un interprete per il linguaggio \( L \) (scritto nel linguaggio \( L_0 \)) è un programma
  \( I^{L_0L}: (Prog^L \times D) \to D \) che prende in input un programma \( P^L \) e un dato
  e restituisce in output il risultato dell'esecuzione di \( P^L \) su \( input \).
  Questa è la realizzazione della \textbf{macchina di Turing universale}.
  \[
    \underbrace{I^{L_0L}(P^L, input) = output \quad t.c. \quad P^L(input) = output}
  \] 
  \[
    I^{L_0L}(P^L, input) = P^L(input) \Rightarrow \text{Macchina universale}
  \] 
\end{definition}

\subsubsection{Struttura dell'interprete}
La struttura dell'interprete è data dal seguente pseudocodice:
\begin{lstlisting}
begin
  go = true;
  while go do
    fetch(opcode, opinfo) at PC; // Program counter
    decode(opcode, opinfo)

    if opcode needs arguments then
      fetch(args)

    case opcode
      opcode1: execute(opcode1, args)
      opcode2: execute(opcode2, args)
      ...
      halt: go = false

    if opcode has result then store(result)

    PC = PC + sizeof(opcode)

end
\end{lstlisting}

\begin{example}
  Un semplice autointerprete, cioè un interprete scritto nel linguaggio che interpreta.
\begin{lstlisting}[language=Python]
input P, d;
pc = 2;
store = [in -> d, out -> 0, x_1 -> 0, x_2 -> 0, ...];
while pc < length(P) do
  # Find the pc-th instruction
  instruction = lookup(P, pc);

  # Syntax rules
  case instruction of
    skip: pc = pc + 1;
    x = e: store = store[x -> eval(e, store)]; pc = pc + 1;
    ...
  endcase
endwhile;

output store[out]


# Semantic rules
eval(e, store) = case e of
  constant: e
  variable: store(e)
  e1 + e2: eval(e1, store) + eval(e2, store)
  e1 - e2: eval(e1, store) - eval(e2, store)
  e1 * e2: eval(e1, store) * eval(e2, store)
  ...
endcase
\end{lstlisting}
Dove \texttt{in -> d} rappresenta l'assegnamento del dato di input alla variabile \texttt{in}.
\end{example}
\noindent
L'interprete \textbf{realizza} la macchina astratta \( M_L \) di un
linguaggio di programmazione \( L \) partendo da un linguaggio \( L_0 \) 
con una macchina astratta \( M_0 \) direttamente eseguibile passando
attraverso una macchina intermedia \( M_I \) (interprete).

\subsection{Compilatore}
\begin{definition}
  Il compilatore da \( L \) a \( L_0 \) è un programma del tipo:
  \[
    C^{LL_0}: Prog^L \to Prog^{L_0}
  \] 
  che prende in input un programma \( P^L \) e restituisce in output un programma \( P^{L_0} \).
  \[
    \forall P^L \in Prog^L \;.\; C^{LL_0}(P^L) = P^{L_0} \quad t.c. \quad
    \forall d \in D \;.\; P^{L_0}(d) = P^L(d)
  \] 
  Non ci sono accessi alla memoria o a dati. Verificare che due programmi
  siano uguali è un problema non decidibile, quindi il compilatore deve
  essere costruito in modo da garantire che i programmi compilati siano
  equivalenti a quelli originali.
\end{definition}

\end{itemize}

\subsubsection{Prima proiezione di Futamura}
Un compilatore può essere ottenuto \textbf{specializzando} un interprete
su un programma.

\begin{definition}[Specializzatore]
  Uno specializzatore per un linguaggio di programmazione \( L \) è un
  programma che realizza la funzione:
  \[
    Spec_L: (Prog^L \times D) \to Prog^L
  \] 
  tale che:
  \[
    \begin{aligned}
      & \forall P^L \in Prog^L, \forall d \in D \;.\; \\
      &\quad Spec_L(P^L, d) = P^L_d\\
      &\quad \; t.c. \; P^L(d, input) = P^L_d(input)
    \end{aligned}
  \] 
  Quindi \( d \) non è più un dato di input, ma è incorporato nel codice.

  \vspace{1em}
  \noindent
  Lo specializzatore fornisce un legame tra interprete e compilatore. Infatti
  specializzando un interprete (scritto in \( L \)) si ottiene un
  compilatore da \( L \) a \( L_0 \).
  \[
    I(P^L, d) \Rightarrow Spec(I, P^L)
  \] 
  \( Spec \) è un programma scritto in \( L_0 \) che integra nel codice
  l'esecuzione di \( P^L \).
\end{definition}

\section{Descrivere i linguaggi di programmazione}
I linguaggi di programmazione sono dei linguaggi, quindi hanno in caratteristiche
in comune con i linguaggi e quindi anche quelli naturali. Le caratteristiche sono:
\begin{itemize}
  \item \textbf{Sintassi}: Regole di formazione. Determinano quali sono
    le frasi ben formate nel linguaggio e descrive relazioni tra simboli

  \item \textbf{Semantica}: Attribuzione del \textbf{significato} ai simboli.
    Determina il significato delle frasi ben formate nel linguaggio e descrive
    relazioni tra simboli e significati.

  \item \textbf{Pragmatica}: Regole di buon utilizzo del linguaggio.

  \item \textbf{Implementazione} (Caratteristica aggiunta per i linguaggi di programmazione):
    Esistono strumenti che eseguono frasi ben formate del linguaggio.
\end{itemize}

\begin{example}
  Il numero è soltanto sintassi, mentre il significato è il valore che assume:
  \[
    4 \quad \text{(sintassi)} \to \text{significato} \quad 4
  \] 
  \[
    IV \quad \text{(sintassi)} \to  \text{significato} \quad 4
  \] 
\end{example}

\subsection{Sintassi}
Le componenti principali della sintassi sono:
\begin{itemize}
  \item \textbf{Parola}: stringa su un alfabeto \( \Sigma \)

  \item \textbf{Frase}: sequenza di parole ben formata, cioè che rispetta le regole
    della sintassi

  \item \textbf{Linguaggio}: insieme di frasi ben formate

    \begin{example}
      Prendiamo ad esempio il seguente linguaggio:
      \[
        \left\{ a^n b^n \;\left|\; n \in \mathbb{N} \right. \right\}
      \] 
      \begin{itemize}
        \item \textbf{Alfabeto}: \( \Sigma = \left\{ a, b \right\} \) 
        \item \textbf{Parole}: \( \varepsilon, a, b, aa, ab, ba, bb, aaa, aab, ... \)
        \item \textbf{Linguaggio}: \(
          \left\{ a^n b^n \;\left|\; n \in \mathbb{N} \right. \right\}
        \)
      \end{itemize}
    \end{example}

  \item \textbf{Lessema}: una parola (costituita da soli terminali nell'alfabeto) che
  ha un significato, cioè è una parola che rappresenta un concetto o un oggetto nel
  linguaggio.

  \begin{example}
    Alcuni esempi di lessemi sono:
    \begin{itemize}
      \item \texttt{if exp then com else com}:\\
        I lessemi sono \texttt{if}, \texttt{then}, \texttt{else}, mentre \texttt{exp}
        e \texttt{com} non sono lessemi perchè rappresentano un'espressione e un comando,
        cioè sono dei simboli non terminali.
    \end{itemize}
  \end{example}

  \item \textbf{Token}: categorie sintattiche, cioè simboli non terminali

    \begin{example}
      Alcuni esempi di token sono:
      \item \texttt{if exp then com else com}:\\
        I token sono \texttt{exp} e \texttt{com} perchè sono simboli non terminali.
    \end{example}
\end{itemize}

\subsubsection{Strumenti di definizione della sintassi}
Gli strumenti per definire la sintassi di un linguaggio di programmazione sono:
\begin{itemize}
  \item \textbf{Riconoscitore}: è uno strumento per il \textbf{riconoscimento} di
    stringhe appartenenti al linguaggio. Ad esempio gli automi a stati finiti, utilizzati
    nell'analisi lessicale per il riconoscimento dei lessemi.

  \item \textbf{Generatore}: è uno strumento per la \textbf{generazione} di stringhe
    appartenenti al linguaggio. Ad esempio le grammatiche CF, utilizzate nella fase
    di parsing della compilazione per riconoscere programmi scritti in un linguaggio
    di programmazione.
\end{itemize}

\begin{define}
  Un linguaggio di programmazione è dipendente dal contesto, quindi si cerca di svilupparli
  "a livelli", cioè si fa prima tutto ciò che si può fare con gli automi, poi tutto ciò
  che si può fare con le grammatiche CF, e così via.
\end{define}

\begin{definition}
  Una grammatica context free è definita da:
  \[
    G = \left<V, T, P, S \right>
  \] 
  dove:
  \begin{itemize}
    \item \( V \): è un insieme finito di simboli non terminali (variabili)
    \item \( T \): è un insieme finito di simboli terminali
      \[
        V \cap T = \varnothing
      \] 
    \item \( P \): è un insieme finito di procedure:
      \[
        A \to \alpha \quad A \in V, \alpha \in (V \cup T)^*
      \] 
    \item \( S \in V \): è il simbolo iniziale
  \end{itemize}
  In un linguaggio di programmazione la grammatica CF è mappata nel seguente modo:
  \begin{itemize}
    \item \( V \): categorie sintattiche (ad esempio le espressioni, le variabili, ecc...)
    \item \( T \): vocabolario di parole (\texttt{if}, \texttt{then}, \texttt{else}, ecc...)
    \item \( P \): regole di formazione
    \item \( S \in V \): categoria della frase (il programma)
  \end{itemize}

  \vspace{1em}
  \noindent

\end{definition}

\begin{define}[Notazione BNF]
  Per i linguaggi di programmazione viene utilizzata la notazione BNF (Backus-Naur Form)
  che cambia i simboli utilizzati per rappresentare i componenti della grammatica CF,
  ad esempio:
  \[
    \to \quad \text{diventa} \quad ::=
  \] 
  Un linguaggio BNF è CF.
\end{define}

\begin{example}
  Costruiamo una grammatica per espressioni booleane:
  \[
    G = \left< \{Exp\}, \{0, 1, or, and, not, (, )\}, P, Exp \right>
  \] 
  \[
    \begin{aligned}
      P &= \left\{
        \begin{aligned}
        & Exp \to 0 \\
        & Exp \to 1 \\
        & Exp \to (Exp \; or \; Exp) \\
        & Exp \to (Exp \; and \; Exp) \\
        & Exp \to (not \; Exp) \\
        \end{aligned}
      \right.\\
        &= 0 | 1 | (Exp \; or \; Exp) | (Exp \; and \; Exp) | (not \; Exp)
    \end{aligned}
  \] 
\end{example}

\subsubsection{Definizione non formale}
Un linguaggio si può definire anche in un modo non formale, definendo
i costrutti base:
\begin{itemize}
  \item Niente dichiarazioni
  \item Solo variabili ed espressioni booleane
  \item Istruzioni:
    \begin{itemize}
      \item Assegnamento
      \item Composizione sequenziale
      \item Comando iterativo (ciclo)
      \item Comando condizionale (if)
    \end{itemize}
\end{itemize}
Il linguaggio è definito su livelli, partendo dal basso verso l'alto:
\begin{enumerate}
  \item Notazione della grammatica

  \item Linguaggio regolare per riconoscere gli identificatori e le parole
    riservate al linguaggio di programmazione

  \item Espressioni booleane che parte dai simboli definiti ai livelli
    sottostanti
    \[
      \color{blue}B\color{black} \to true \mid false \mid
      (\color{blue}B\color{black} \; or \; \color{blue}B\color{black}) \mid
      (\color{blue}B\color{black} \; and \; \color{blue}B\color{black}) \mid
      (not \; \color{blue}B\color{black}) \mid x
    \] 

  \item Comandi atomici, ad esempio l'assegnamento. Utilizza gli identificatori,
    un simbolo di assegnamento e un espressione (booleana in questo caso).
    \[
      \color{green}A\color{black} \to x := \color{blue}B\color{black}
    \] 

  \item Comandi, ad esempio l'if o il ciclo. Utilizzano i comandi atomici
    e le espressioni booleane.

    \[
      \color{purple}C\color{black} \to \color{green}A\color{black} \mid
      \color{purple}C\color{black} ; \color{purple}C\color{black} \mid
      while \; \color{blue}B\color{black} \; do \; \color{purple}C\color{black} \mid
      if \; \color{blue}B\color{black} \; then \color{purple}C\color{black} \; else
      \; \color{purple}C\color{black}
    \] 

  \item Programmi, cioè sequenze di comandi.
    \[
      \color{orange}P\color{black} \to
      \left\{ \color{purple}C\color{black} \right\}
    \] 
\end{enumerate}

  \centering
  \includegraphics[width=0.8\textwidth]{livelli}
  \caption{Rappresentazione grafica dei livelli di un linguaggio di programmazione}
\end{figure}
Si utilizzano espressioni parentesizzate per evitare ambiguità. Se non
si utilizzassero le parentesi si potrebbe scegliere di eseguire prima
un'operatore invece di un altro e questo porterebbe a risultati diversi
dalla stessa espressione.


\subsection{Analisi semantica}
L'analisi semantica verifica i \textbf{vincoli sintattici}, ad esempio che il numero
di parametri formali sia uguale al numero di parametri attuali, oppure
che l'uso di una variabile sia possibile solo dopo che sia stata dichiarata.
Questi sono vincoli sul codice, quindi \textbf{non catturati dalla grammatica} e si
dicono \textbf{contestuali}.
\begin{itemize}
  \item Sintassi \( \to \) Grammatica CF
  \item Semantica \( \to \) Tutti i processi di verifica ed esclusione che non sono
    descritti dalla sintassi
\end{itemize}

\subsubsection{Induzione strutturale}
L'induzione strutturale permette di dimostrare proprietà di oggetti descritti
in modo induttivo o ricorsivo. Una grammatica descrive in modo induttivo il
linguaggio (insieme di stringhe) che genera, quindi si può utilizzare l'induzione
strutturale per dimostrare proprietà delle stringhe generate.

\begin{definition}
  Per dimostrare una proprietà \( \Pi \) su tutti gli elementi di un insieme
  definito in modo strutturale da una grammatica bisogna:
  \begin{enumerate}
    \item (Caso base): Dimostrare la proprietà su tutti gli elementi base, cioè quelli
      che sono esplicitamente inseriti nell'insieme di stringhe generate dalla
      grammatica.

    \item (Passo induttivo): Dimostrare la proprietà sugli elementi composti sotto \textbf{l'ipotesi
      induttiva} che la proprietà valga per gli elementi che la compongono
  \end{enumerate}
\end{definition}

\vspace{1em}
\noindent

\begin{example}
  Definiamo l'insieme \( X \subseteq \mathbb{N} \):
  \[
  \begin{cases}
    \text{Base} &: 0 \in X \\
    \text{Passo induttivo} &: \text{se } n \in X \text{ allora } n + 3 \in X
  \end{cases}
  \] 
  La proprietà è \( X = 3 \mathbb{N} \) 

  \vspace{1em}
  \noindent
  La proprietà dimostrabile in modo strutturale è: \( X \subseteq 3 \mathbb{N} \):
  \begin{enumerate}
    \item \textbf{Caso base}: \( 0 \) è la base di \( X \) e \( 0 \in 3 \mathbb{N} \) 
    \item \textbf{Passo induttivo}: dimostriamo \( \forall n.\; n \in X \Rightarrow n \in 3 \mathbb{N} \).\\
      Prendiamo \( n \in X \), l'ipotesi induttiva è la seguente:
      \[
        n \in X \Rightarrow n \in 3 \mathbb{N}
      \] 
      bisogna dimostrare che \( n+3 \in 3 \mathbb{N} \). Quindi sappiamo che
      \[
        \begin{aligned}
          n \in  3 \mathbb{N} &\Rightarrow \exists i .\; n = 3i\\
                              &\Rightarrow n + 3 = 3i + 3 = 3(i + 1) \in 3 \mathbb{N}
        \end{aligned}
      \] 
  \end{enumerate}
\end{example}

\begin{example}
  Definiamo induttivamente gli alberi binari in cui i nodi sono numeri \( n \in \mathbb{N} \).
  Ogni numero è un albero binario, perchè è un albero con un solo nodo:
  \[
    n \in BT \quad \text{(Base)}
  \] 
  Consideriamo due alberi binari \( T_1 \in BT \) e \( T_2 \in BT \) allora:
  \begin{figure}[H]
    \centering
    \begin{forest}
      [n
        [\( T_1 \)]
        [\( T_2 \)]
      ]
    \end{forest}
    \( \in BT \) 
    \caption{Albero binario composto da due alberi binari}
  \end{figure}
  Quindi:
  \[
    br(T_1, T_2) \in BT
  \] 
  dove \( br() \) è l'operatore che aggiunge un nodo una radice comune a \( T_1 \) 
  e \( T_2 \) 


  \vspace{1em}
  \noindent
  \textbf{IMPORTANTE:} Dimostriamo che la funzione che calcola il numero di foglie
  (\texttt{\#foglie(T)}) e la funzione che restituisce il numero di nodi che
  hanno figli (\texttt{\#branch(T)}) sono uguali a:
  \[
    \forall T \in BT.\; \texttt{\#foglie}(T) = \texttt{\#branch}(T) + 1
  \] 
  Definiamo la funzione \texttt{\#foglie(T)}:
  \[
    \left\{
    \begin{aligned}
      &\#foglie(n) = 1 \\
      &\#foglie(br(T_1, T_2)) = \#foglie(T_1) + \#foglie(T_2)
    \end{aligned}
    \right.
  \] 
  Definiamo la funzione \texttt{\#branch(T)}:
  \[
    \left\{
    \begin{aligned}
      &\#branch(n) = 0 \\
      &\#branch(br(T_1, T_2)) = \#branch(T_1) + \#branch(T_2) + 1
    \end{aligned}
    \right.
  \]
  Dimostrazione per induzione sulla costruzione degli elementi di \( BT \):
  \begin{enumerate}
    \item \textbf{Caso Base}: è il nodo che non ha figli in BT:
      \[
        T = n \in BT
      \] 
      Sappiamo che:
      \[
        \#foglie(T) = \#foglie(n) = 1 = 1 + 0 = 1 + \#branch(T)
      \] 
      e questo soddisfa la proprietà.

    \item \textbf{Passo induttivo}: \( T = br(T_1, T_2) \). Ipotesi
      induttiva: supponiamo che \( T_1 \) e \( T_2 \) soddisfino la proprietà,
      quindi:
      \[
        \begin{aligned}
          \#foglie(T_1) &= \#branch(T_1) + 1 \\
          \#foglie(T_2) &= \#branch(T_2) + 1
        \end{aligned}
      \] 
      Dobbiamo dimostrare che \( \#foglie(T) = \#branch(T) + 1 \):
      \[
        \begin{aligned}
          \#foglie(T) &\stackrel{def\; T}{=} \#foglie(br(T_1, T_2)) \\
                      &\stackrel{def \; \#foglie}{=} \#foglie(T_1) + \#foglie(T_2) \\
                      &\stackrel{Hp\; Ind}{=}
                  \underbrace{\#branch(T_1) + 1 + 
                  \#branch(T_2)}_{def\; \#branch(br(T_1, T_2))}
                  + 1 \\
                      &= \#branch(\underbrace{br(T_1, T_2)}_{T}) + 1 \\
                      &= \#branch(T) + 1
        \end{aligned}
      \] 
      Quindi anche \( T \) soddisfa la proprietà.
  \end{enumerate}
\end{example}

\begin{example}
  Consideriamo la seguente grammatica:
  \[
    S \to aa \mid bb \mid aSa \mid bSb
  \] 
  questa grammatica genera tutte le stringhe palindrome di lunghezza pari.
  Siccome \( aa \) e \( bb \) sono nel linguaggio, se si prende una qualsiasi
  stringa nel linguaggio \( S \) e si aggiungono \( a \) o \( b \) all'inizio e
  alla fine, si ottiene una stringa ancora palindroma, quindi anche questa stringa è
  nel linguaggio. Si può quindi definire questa grammatica in modo induttivo:
  \[
    LS: 
    \begin{cases}
      aa, bb, \in LS \quad \text{(Base)} \\
      \sigma \in LS \Rightarrow a\sigma a, b\sigma b \in LS \quad \text{(Passo induttivo)}
    \end{cases}
  \] 
  Quindi \( L(S) = LS \) e la proprietà da dimostrare è:
  \[
    \sigma \in LS \Rightarrow \exists x, y.\; \sigma = x y \land x = reverse(y)
  \] 

  \begin{enumerate}
    \item \textbf{Caso base}: Prendiamo:
      \[
        aa = \sigma \Rightarrow x = a, y = a \quad \text{e} \quad
        x = a = reverse(a) = reverse(y)
      \] 
      analogo per \( \sigma = bb \).

    \item \textbf{Passo induttivo}: Prendiamo \( \sigma \in LS \), l'ipotesi
      induttiva è la seguente:
      \[
        \sigma = xy \;t.c.\; x = reverse(y)
      \] 
      Prendiamo \( \sigma' = a \sigma a \). Per ipotesi induttiva:
      \[
        \sigma' = a \sigma a = a \overbrace{xy}^{\sigma} a = x'y'
      \] 
      con \( x' = ax \) e \( y' = ya \). Quindi:
      \[
        \begin{aligned}
          x' = ax &= a reverse(y) \\
                  &= reverse(\underbrace{ya}_{y'}) \\
                  &= reverse(y')
        \end{aligned}
      \] 
      Abbiamo dimostrato che \( \sigma' \) è palindroma, quindi \( \sigma' \in LS \).
  \end{enumerate}
\end{example}

\begin{exercise}
  Consideriamo la seguente grammatica:
  \[
    E \to n \mid E + E \mid E * E
  \] 
  \begin{enumerate}
    \item Definire strutturalmente l'insieme \( Exp \)

    \item Si considerino le seguenti funzioni:
      \begin{itemize}
        \item Conta il numero di operatori:\\
          \( op(E): op(n) = 0 \quad op(E_1 + E_2) = op(E_1 * E_2) = 1 + op(E_1) +
          op(e_2) \)

        \item Conta il numero di valori:\\
          \( val(E): val(n) = 1 \quad val(E_1 + E_2) = val(E_1 * E_2) = val(E_1) + val(E_2) \)
      \end{itemize}
      Dimostrare che:
      \[
        \forall E \;.\; val(E) = op(E) + 1
      \] 
  \end{enumerate}

  Definizione strutturale dell'insieme \( Exp \):
  \begin{itemize}
    \item \textbf{Base}: \( n \in L(E) \)
    \item \textbf{Passo induttivo}: se \( e_1, e_2 \in Exp \) allora
      \( e_1 + e_2, e_1 * e_2 \in Exp \).
      Le espressioni sono il minimo insieme chiuso che soddisfa queste condizioni,
      quindi non ci sono altre espressioni in \( L(E) \). Quindi ad esempio:
      \[
        3 + 5 * 2 \in Exp
      \] 
  \end{itemize}
  Sappiamo che:
  \[
    \begin{aligned}
      &\begin{cases}
        val(n) = 1 \\
        val(e_1 + e_2) = val(e_1 * e_2) = val(e_1) + val(e_2)
      \end{cases} \\
      &\begin{cases}
        op(n) = 0 \\
        op(e_1 + e_2) = op(e_1 * e_2) = op(e_1) + op(e_2) + 1
      \end{cases}
    \end{aligned}
  \] 
  Quindi sull'esempio precedente:
  \[
    \begin{aligned}
      val(\underbrace{3 + 5}_{e_1} * \underbrace{2}_{e_2}) &= val(3 + 5) + val(2) \\
                                                           &= val(3) + val(5) + val(2) \\
                                                           &= 1 + 1 + 1 = 3
    \end{aligned}
  \] 
  \[
    \begin{aligned}
      op(\underbrace{3 + 5}_{e_1} * \underbrace{2}_{e_2}) &= op(3 + 5) + op(2) + 1 \\
                                                          &= op(3) + op(5) + op(2) + 1 + 1 \\
                                                          &= 0 + 0 + 0 + 1 + 1 = 2
    \end{aligned}
  \] 
  Dimostriamo che \( \Pi:\; \forall e \in Exp \;.\; val(e) = op(e) + 1 \) per induzione
  strutturale su \( Exp \) 
  \begin{itemize}
    \item \textbf{Caso base}: Verifichiamo \( \Pi  \) sugli elementi esplicitamente
      inseriti nell'insieme \( Exp \), cioè sui numeri \( n \in \mathbb{N}\):
      consideriamo le espressioni \( e = n \) 
      \[
        val(e) = val(n) = 1 \quad \textbf{=} \quad 1 + 0 = 1 + op(n) = 1 + op(e)
      \] 

    \item \textbf{Ipotesi induttiva}: Dimostriamo per gli elementi composti supponendo
      l'ipotesi induttiva sugli elementi che li compongono. Sia \( e = e_1 + e_2 \) (ipotesi1)
      (analogo per \( e = e_1 * e_2 \)), l'ipotesi induttiva è la seguente:
      \[
        val(e_1) = op(e_1) + 1 \quad \land \quad val(e_2) = op(e_2) + 1
      \] 
      Dobbiamo dimostrare che \( val(e) = op(e) + 1 \):
      \[
        \begin{aligned}
          val(e) &\stackrel{ipotesi1}{=} val(e_1 + e_2) \\
                 &\stackrel{def\; val}{=} val(e_1) + val(e_2) \\
                 &\stackrel{Hp.\; Ind.}{=} \underbrace{op(e_1) + 1 + op(e_2)}_{op(e_1 + e_2)} + 1 \\
                 &\stackrel{def\; op}{=} op(e_1 + e_2) + 1 \\
                 &\stackrel{ipotesi1}{=} op(e) + 1
        \end{aligned}
      \] 
  \end{itemize}
  Abbiamo dimostrato che \( \Pi \) è vera per tutti gli elementi di \( Exp \).
\end{exercise}

\subsubsection{Semantica}
La semantica è l'attribuzione di significato ai simboli.
Il significato è un'entità autonoma che esiste indipendentemente dai simboli
usati per descriverlo. Esistono 3 tipi di semantica:
\begin{itemize}
  \item \textbf{Denotazionale}: È un modello completamente matematico che descrive il
    programma come una funzione su stati (programmi come funzioni). Lo \textbf{stato}
    descrive il valore di tutte le variabili in un certo momento.
    \[
      Stato: Var \mapsto Val
    \] 
    La semantica denotazionale è una funzione che prende un programma e restituisce
    una funzione che prende uno stato iniziale e restituisce uno stato finale:
    \[
      E: Prog \mapsto \underbrace{(Stato \to Stato)}_{\substack{\text{Denotazione associata}\\\text{al programma}}}
    \] 
    Un esempio di semantica denotazionale è il comportamento input/output

  \item \textbf{Operazionale}: Descrive il significato del programma come sequenza
    di trasformazione di stati, \textbf{eseguendo} i suoi comandi sulla macchina astratta.
    Per semplificare vedremo lo stato come se fosse la memoria:
    \[
      \text{Memoria: } \sigma: Var^n \mapsto Val^n
    \] 
    \[
      \text{Stato: } (Prog \times Mem) \cup Mem
    \] 
    dove:
    \begin{itemize}
      \item \( Prog \) è il programma che stiamo eseguendo
      \item \( Mem \) è la memoria su cui stiamo eseguendo il programma
      \item \( \cup Mem \) è l'insieme degli stati finali
    \end{itemize}
    Ad esempio:
    \[
      \text{Variabili:} <x, y, z> \mapsto <2, 5, 3>
      \leadsto <x \mapsto 2, y \mapsto 5, z \mapsto 3>
    \] 
    Per eseguire un programma si utilizza un \textbf{sistema di transizione}:
    \[
      <\Sigma, \to>
    \] 
    dove:
    \begin{itemize}
      \item \( \Sigma \) è l'insieme degli stati
      \item \( \to \) è la relazione di transizione che descrive come si passa da
        uno stato ad un altro eseguendo un comando del programma
    \end{itemize}

  \item \textbf{Assiomatica}: Descrive il programma attraverso le proprietà che
    soddisfa. Si calcola utilizzando un sistema di derivazione che fornisce regole per
    dimostrare proprietà sui programmi:
    \begin{figure}[H]
      \centering
      \includegraphics[width=0.6\textwidth]{assiomatica}
      \caption{Esempio di sistema di derivazione per la semantica assiomatica}
    \end{figure}
    \noindent
    Quindi descrive relazioni tra proprietà di stati.
    Ad esempio fornendo le proprietà invarianti dei programmi, cioè le
    proprietà che sono vere prima e dopo l'esecuzione di un programma.
\end{itemize}

\begin{example}
  Consideriamo il seguente programma:
  \[
  P = \begin{cases}
    z := 2; \\
    y := z; \\
    y := y + 1; \\
    z := y;
  \end{cases}
  \] 
  Applichiamo i tre tipi diversi di semantica al programma \( P \):
  \begin{itemize}
    \item \textbf{Semantica denotazionale}:
      \[
        \begin{aligned}
          &\mathcal{D}[\overbrace{P}^{Prog}]
          (\overbrace{z \mapsto \bot, y \mapsto \bot}^{Stato\; iniziale}) = \\
          &= \mathcal{D}[z := 2; y := z; y := y + 1; z := y](z \mapsto \bot, y \mapsto \bot) \\
          &= \mathcal{D}[z := y] \circ \mathcal{D}[y := y + 1] \circ \mathcal{D}[y := z] \circ
          \color{blue}\mathcal{D}[z := 2] (z \mapsto \bot, y \mapsto \bot) \\
          &= \mathcal{D}[z := y] \circ \mathcal{D}[y := y + 1] \circ
          \color{blue}\mathcal{D}[y := z] (z \mapsto 2, y \mapsto \bot) \\
          &= \mathcal{D}[z := y] \circ
          \color{blue}\mathcal{D}[y := y + 1] (z \mapsto 2, y \mapsto 2) \\
          &= \color{blue}\mathcal{D}[z := y] (z \mapsto 2, y \mapsto 3) \\
          &= \color{blue}(z \mapsto 3, y \mapsto 3)
        \end{aligned}
      \] 
      Quindi \( \mathcal{D}[P](z \mapsto \bot, y \mapsto \bot) = (z \mapsto 3, y \mapsto 3) \)
      è usata per:
      \begin{itemize}
        \item Dimostrare proprietà (correttezza)
        \item Fornisce uno strumento formale per ragionare sulle funzionalità dei
          programmi
        \item Non è utile a chi usa il linguaggio di programmazione
      \end{itemize}

    \item \textbf{Semantica assiomatica}: Fornisce proprietà relazionali sull'input
      e sull'output del programma, ad esempio:
      \begin{figure}[H]
        \centering
        \includegraphics[width=0.6\textwidth]{esempio_assiomatica}
        \caption{Esempio di proprietà assiomatica per il programma \( P \)}
      \end{figure}

    \item \textbf{Semantica operazionale}:
      \[
        \begin{aligned}
          &<P, (z \mapsto \bot, y \mapsto \bot)> = \\
          &= <\color{blue}z := 2\color{black}; y := z; y := y + 1; z := y, (z \mapsto \bot, y \mapsto \bot)> \\
          &= <\color{blue}y := z\color{black}; y := y + 1; z := y, (z \mapsto 2, y \mapsto \bot)> \\
          &= <\color{blue}y := y + 1\color{black}; z := y, (z \mapsto 2, y \mapsto 2)> \\
          &= <\color{blue}z := y\color{black}, (z \mapsto 2, y \mapsto 3)> \\
          &= <z \mapsto 3, y \mapsto 3>
        \end{aligned}
      \] 
  \end{itemize}
\end{example}

\subsubsection{Composizionalità della semantica}
Il significato di ogni programma è funzione del significato dei suoi componenti
immediati, cioè si possono ottenere le proprietà semantiche di un programma a partire
dalle proprietà semantiche dei suoi componenti.

\subsubsection{Equivalenza di programmi}
La semantica viene utilizzata per verificare l'equivalenza tra programmi.
\begin{itemize}
  \item \( P_1 = P_2 \) dice che \( P_1 \) è esattamente lo stesso programma di \( P_2 \),
    cioè sono sintatticamente uguali. Questa si dice \textbf{uguaglianza sintattica}.

  \item \( P_1 \equiv P_2 \) dice che \( P_1 \) e \( P_2 \) hanno lo stesso comportamento
    (stesso significato), anche se sono scritti in modo diverso. Questa si dice
    \textbf{equivalenza} o uguaglianza semantica.
\end{itemize}

\begin{example}
  Consideriamo i seguenti programmi:
  \begin{itemize}
    \item \( P_1 \) insertion sort
    \item \( P_2 \) bubblue sort
  \end{itemize}
  Sappiamo che i due programmi sono diversi, quindi:
  \[
    P_1 \neq P_2
  \] 
  Ma sappiamo che entrambi ordinano una lista, quindi il loro comportamento è lo stesso:
  \begin{itemize}
    \item Con la semantica denotazionale si ha:
      \[
        \mathcal{D}[P_1] = \mathcal{D}[P_2] \leadsto P_1 \equiv P_2
      \] 

    \item Con la semantica operazionale si ha:
      \[
        [\![ P_1 ]\!] \neq [\![ P_2 ]\!] \leadsto P_1 \not\equiv P_2
      \] 
  \end{itemize}
  Quindi l'equivalenza dipende dalla scelta della semantica
\end{example}

\subsubsection{Categorie sintattiche}
Le categorie sintattiche sono insiemi di stringhe che condividono una certa proprietà
per:
\begin{itemize}
  \item Manipolare dati (espressioni)
  \item Trasformare stati (comandi)
  \item Creare e manipolare i legami (dichiarazioni), ad esempio variabile-valore,
    variabile-tipo, funzione-codice, ecc...
\end{itemize}
Si hanno quindi:
\begin{itemize}
  \item \textbf{Stato}: descrive gli effetti sugli identificatori
  \item \textbf{Ambiente}: è un insieme di legami tra identificatori e oggetti
    denotati
\end{itemize}

\vspace{1em}
\noindent
Le categorie sintattiche principali sono:
\begin{itemize}
  \item \textbf{Espressioni}: denotano e manipolano valori. Le espressioni possono
    essere complesse e quindi devono essere \textbf{valutate} per restituire il
    singolo valore che sta denotando.
    Espressioni \textbf{sintatticamente diverse} possono denotare lo stesso valore.
    Ad esempio \( 1 + 5 \) e \( 3 * 2 \) denotano lo stesso valore.

    \begin{definition}
      Due espressioni sono equivalenti se vengono valutate nello \textbf{stesso}
      valore a partire da qualunque stato/memoria iniziale.
    \end{definition}


  \item \textbf{Comandi}: denotano richieste di modifica della memoria.
    \begin{itemize}
      \item Le strutture di controllo decidono quali comandi eseguire
      \item Gli assegnamenti modificano effettivamente la memoria
    \end{itemize}
    Rappresentano funzioni: \( Memoria \mapsto Memoria \). I comandi vengono
    \textbf{eseguiti} per attuare la corrispondente richiesta di trasformazione della
    memoria. Queste trasformazioni sono \textbf{irreversibili}, cioè non possono essere
    cancellate, ma solo annullate semanticamente, ad esempio:
    \[
      x := x + 1; \; x := x - 1
    \] 
    Questi comandi si annullano, ma lo stato è stato comunque modificato.

    \begin{definition}
      Due comandi sono equivalenti se per ogni stato iniziale in input producono lo
      stesso stato in output. Quindi l'equivalenza è vista come uguaglianza della
      funzione calcolata.
    \end{definition}

  \item \textbf{Dichiarazioni}: denotano richieste di creazione o modifica di legami,
    ovvero di \textbf{ambienti}. Le dichiarazioni vengono \textbf{elaborate} per
    attuare le richieste di creazione o modifica degli ambienti.

    \begin{definition}
      Due dichiarazioni sono equivalenti se producono lo stesso ambiente (e la stessa
      memoria) a partire da qualunque stato iniziale.
    \end{definition}
\end{itemize}
La separazione tra le categorie sintattiche non è rigida, ad esempio un'espressione
può avere anche side effects, quindi modificare la memoria, ecc...

\subsubsection{Esempio di sintassi per un linguaggio giocattolo "PLO"}
In questo corso considereremo un linguaggio di programmazione "giocattolo" 
simile al linguaggio Pascal in cui:
\begin{itemize}
  \item Esiste un singolo tipo di dato \texttt{integer}
  \item I booleani sono rappresentati come interi
  \item Strutture di controllo standard
  \item Astrazione del controllo
  \item Le procedure non hanno parametri
\end{itemize}
Il linguaggio è definito nel seguente modo:
\[
  \begin{aligned}
    \texttt{\color{orange}<Program>} \quad &\texttt{\color{orange}P\color{black}} \to
    \texttt{\color{purple}B} \\
    %
    \texttt{\color{purple}<Block>} \quad &\texttt{\color{purple}B\color{black}} \to
    \texttt{\color{Green}D\color{blue}S} \\
    %
    \texttt{\color{Green}<Declar>} \quad & \texttt{\color{Green}D\color{black}} \to
    \texttt{[const I = N\{,I=N\};]} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{[var I\{,I\};]} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{[procedure I;\color{purple}B\color{black};]} \\
    %
    \texttt{\color{blue}<Stmts>} \quad & \texttt{\color{blue}S\color{black}} \to
    \varepsilon 
    \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{I := \color{gray}E} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{call I} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{if \color{olive}C\color{black}\ then \color{blue}S\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{while \color{olive}C\color{black}\ do \color{blue}S\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{begin \color{blue}S\color{black}\{;\color{blue}S\color{black}\} end} \\
    %
    \texttt{\color{olive}<Cond>} \quad & \texttt{\color{olive}C\color{black}} \to
    \texttt{odd \color{gray}E\color{black}} 
                    \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ > \color{gray}E\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ >= \color{gray}E\color{black}} 
                    \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ = \color{gray}E\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ \# \color{gray}E\color{black}} 
                    \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ < \color{gray}E\color{black}} \\
                   & \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ <= \color{gray}E\color{black}} \\
    %
    \texttt{\color{gray}<Expr>} \quad & \texttt{\color{gray}E\color{black}} \to
    \texttt{I} \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{N} 
                      \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{\color{gray}E\color{black}\ op \color{gray}E\color{black}} 
                      \color{BrickRed}\color{BrickRed}\mid\color{black}\color{black} \texttt{(\color{gray}E\color{black})}
  \end{aligned}
\] 
\begin{itemize}
  \item I programmi \( \color{orange}P \) sono blocchi \( \color{purple}B \).
  \item I blocchi \( \color{purple}B \) sono una dichiarazione \( \color{Green}D \)
    seguita da un comando \( \color{blue}S \).
    Si noti che \( \color{Green}D \) ed \( \color{blue}S \) sono definiti in modo
    libero dal contesto, ovvero quello che possiamo generare da \( \color{blue}S \)
    non deve avere necessariamente nessun vincolo con quello che generiamo da
    \( \color{Green}D \). In altre parole questa grammatica non ci obbliga in nessun
    modo ad usare identificatori precedentemente definiti. Quindi dalla descrizione
    sintattica rimangono fuori gli aspetti context sensitive (vincoli sintattici)
    che la grammatica CF non è in grado di rappresentare. Servirà la definizione di
    una “semantica statica” che avrà il ruolo di verificare questi vincoli prima di
    compilare e/o eseguire.

  \item Le dichiarazioni \( \color{Green}D \) sono dichiarazioni di valori costanti
    con nome (\texttt{const}), dichiarazioni di variabili (\texttt{var}) e
    dichiarazioni di procedure (ovvero di un blocco riferibile con un nome).

  \item I nomi sono gli identificatori \( I \), considerati qui parte del vocabolario;
    \( N \) sono i numeri naturali, considerati anch’essi parte del vocabolario.

  \item Non abbiamo bisogno del concetto di tipo in quanto abbiamo solo un tipo di dato.

  \item Si noti che le produzioni di \( \color{Green}D \) sono tutte opzionali, questo
    significa che un programma può essere un blocco senza dichiarazioni.

  \item I comandi sono: il comando vuoto, l’assegnamento di un valore (denotato da una
    espressione) ad un identificatore, la chiamata ad una procedura mediante il suo
    nome, il comando condizionale che esegue un comando al verificarsi di una
    condizione booleana \( \color{olive}C \), il ciclo che ripete un comando finché una data
    condizione \( \color{olive}C \) è vera e la composizione sequenziale di comandi.

  \item Le condizioni \( \color{olive}C \) sono espressioni dal valore booleano:
    \texttt{odd} è un predicato unario che verifica se l’espressione è
    0, poi ci sono vari operatori di confronto di espressioni \( \color{gray}E \).

  \item Le espressioni \( \color{gray}E \) sono identificatori, valori naturali,
    operazioni tra espressioni e espressioni tra parentesi.

  \item Si sintetizza con \( \texttt{op} \) la metavariabile dell’insieme
    \( \{+, -, *, /\} \). La stessa cosa si potrebbe fare per gli operatori
    di confronto, mettendo in \( \color{olive}C \) la produzione \( E \; cop \; E \) con
    \( cop \) metavariabile per gli elementi in \( \{ >, >=, =, \#, <=, < \} \).
\end{itemize}
La grammatica delle espressioni è ambigua, ma più comoda da usare e si può utilizzare
per definire la semantica. Per il compilatore però bisogna fornire una grammatica
equivalente precisa e quindi non ambigua, come la seguente:
\[
  \begin{aligned}
    \texttt{\color{gray}<Expr>} \quad &\texttt{\color{gray}E\color{black}} \to
    \texttt{[+,-]\color{gray}
    T\color{black} [\color{gray}A\color{black} \color{gray}T\color{black}]} \\
    %
    \texttt{\color{gray}<Add op>} \quad &\texttt{\color{gray}A\color{black}} \to
    \texttt{+} \color{BrickRed}\mid\color{black} \texttt{-} \\
    %
    \texttt{\color{gray}<Terms>} \quad &\texttt{\color{gray}T\color{black}} \to
    \texttt{\color{gray}F\color{black} [\color{gray}M\color{black}
    \color{gray}F\color{black}]} \\
                                       &\texttt{\color{gray}M\color{black}} \to
                                       \texttt{*} \color{BrickRed}\mid\color{black}
                                       \texttt{/} \\
    %
    \texttt{\color{gray}<Factor>} \quad &\texttt{\color{gray}F\color{black}} \to 
    \color{MidnightBlue}\texttt{I}\color{black} \color{BrickRed}\mid\color{black}
    \color{OliveGreen}\texttt{N}\color{black} \color{BrickRed}\mid\color{black} \texttt{\color{black}(\color{gray}E\color{black})} \\
    %
    \texttt{\color{OliveGreen}{<Numbers>}} \quad &\texttt{\color{OliveGreen}N\color{black}}
    \to \texttt{[+,-]\color{Orchid}d\color{black}\{\color{Orchid}d\color{black}\}} \\
    %
    \texttt{\color{Orchid}{<Digit>}} \quad &\texttt{\color{Orchid}d\color{black}} \to
    \texttt{0} \color{BrickRed}\mid\color{black} \ldots \color{BrickRed}\mid\color{black} \texttt{9} \\
    %
    \texttt{\color{MidnightBlue}{<Ident>}} \quad &\texttt{\color{MidnightBlue}I\color{black}} \to
    \texttt{\color{Maroon}l\color{black}\{\color{Maroon}l\color{black},
    \color{Orchid}d\color{black}\}} \\
    %
    \texttt{\color{Maroon}{<Letter>}} \quad &\texttt{\color{Maroon}l\color{black}} \to
    \texttt{A} \color{BrickRed}\mid\color{black} \ldots \color{BrickRed}\mid\color{black} \texttt{Z}
  \end{aligned}
\] 


\section{Implementazione di un linguaggio di programmazione IMP}

\subsection{Espressioni}

\subsubsection{Aspetti implementativi}
Gli aspetti progettuali da definire per le espressioni sono:
\begin{itemize}
  \item \textbf{Arietà}: cioè a quanti argomenti un operatore si applica

  \item \textbf{Notazione}:
    \begin{itemize}
      \item \textbf{Infissa}: \texttt{e1 + e2}\\ Questa notazione richiede 
        la definizione di regole di \textbf{associatività} e \textbf{precedenza}
        per descrivere quale operatore applicare prima. Le seguentinotazioni non
        ne hanno bisogno.


      \item \textbf{Prefissa}: \texttt{+ e1 e2}\\
        Si utilizza una variabile contatore \texttt{arietà} che tiene traccia del
        numero di operandi dell'operatore da applicare e uno stack in cui
        vengono messi gli operandi.
        \begin{example}
          Consideriamo la seguente espressione:
          \[
             + \, * \, 3 \, 2 \, 5
          \] 
          dove:
          \begin{itemize}
            \item \( * \) accetta due operandi
            \item \( + \) accetta due operandi
          \end{itemize}
          Si inizia con uno stack vuoto e \( \texttt{arietà} = 0 \), quindi si legge da
          sinistra a destra:
          \begin{enumerate}
            \item Si legge \( + \), è un operatore, quindi si mette nello stack e si
              aggiorna \( \texttt{arietà} \):
              \[
                Stack = [+], \quad \texttt{arietà} = 2
              \]
            \item Si legge \( * \), è un operatore, quindi si mette nello stack.
              L'\texttt{arietà} rimane a 2 perchè fa riferimento all'ultimo operatore
              inserito nello stack:
              \[
                Stack = [+, *], \quad \texttt{arietà} = 2
              \]
            \item Si legge \( 3 \), è un operando, quindi si mette nello stack e si
              aggiorna \( \texttt{arietà} \):
              \[
                Stack = [+, *, 3], \quad \texttt{arietà} = 1
              \]
            \item Si legge \( 2 \), è un operando, quindi si mette nello stack e si
              aggiorna \( \texttt{arietà} \):
              \[
                Stack = [+, *, 3, 2], \quad \texttt{arietà} = 0
              \]
              A questo punto \( \texttt{arietà} = 0 \) quindi si applica l'operatore
              \( * \) agli ultimi due operandi inseriti nello stack, si mette il
              risultato nello stack e si aggiorna \( \texttt{arietà} \):
              \[
                Stack = [+, 6], \quad \texttt{arietà} = 1
              \]
            \item Si legge \( 5 \), è un operando, quindi si mette nello stack e
              si aggiorna \( \texttt{arietà} \):
              \[
                Stack = [+, 6, 5], \quad \texttt{arietà} = 0
              \]
              A questo punto \( \texttt{arietà} = 0 \) quindi si applica l'operatore
              \( + \) agli ultimi due operandi inseriti nello stack, si mette il
              risultato nello stack e si aggiorna \( \texttt{arietà} \):
              \[
                Stack = [11], \quad \texttt{arietà} = 0
              \]
          \end{enumerate}
        \end{example}

      \item \textbf{Postfissa}: \texttt{e1 e2 +}\\ Si utilizza uno stack
        che conterrà gli operandi di ciascun operatore, quindi quando si incontra
        un operatore si prendono dal stack gli operandi necessari.
        \begin{example}
          Consideriamo la seguente espressione:
          \[
            5 \, 3 \, 2 \, * \, + 
          \] 
          dove:
          \begin{itemize}
            \item \( * \) accetta due operandi
            \item \( + \) accetta due operandi
          \end{itemize}
          Si inizia con uno stack vuoto, quindi si legge da sinistra a destra:
          \begin{enumerate}
            \item Si legge \( 5 \), è un operando, quindi si mette nello stack:
              \[
                Stack = [5]
              \]
            \item Si legge \( 3 \), è un operando, quindi si mette nello stack:
              \[
                Stack = [5, 3]
              \]
            \item Si legge \( 2 \), è un operando, quindi si mette nello stack:
              \[
                Stack = [5, 3, 2]
              \]
            \item Si legge \( * \), è un operatore, quindi si prendono dal stack
              i due operandi necessari, si applica l'operatore e si mette il
              risultato nello stack:
              \[
                Stack = [5, 6]
              \]
            \item Si legge \( + \), è un operatore, quindi si prendono dal stack
              i due operandi necessari, si applica l'operatore e si mette il
              risultato nello stack:
              \[
                Stack = [11]
              \]
          \end{enumerate}
        \end{example}
    \end{itemize}

  \item \textbf{Regole di associatività}: decidono l'ordine di applicazione di
    operatori allo stesso livello di precedenza. Solitamente per gli operatori
    aritmetici si sceglie l'associatività a sinistra, quindi si applica prima
    l'operatore più a sinistra.


  \item \textbf{Regole di precedenza}: cioè in che ordine valutare le espressioni
    quando non sono esplicitamente raggruppate da parentesi.

    Per gli operatori aritmetici le \textbf{precedenze} sono:
    \begin{enumerate}
      \item \( +, - \) sono allo stesso livello
      \item \( *, / \) sono allo stesso livello e hanno precedenza su \( +, - \)
    \end{enumerate}

   \item \textbf{Ordine di valutazione degli operandi}: cioè in che ordine valutare
      gli operandi di un operatore. Bisogna essere consapevoli delle seguenti
      problematiche legate all'ordine di valutazione degli operandi:
      \begin{itemize}
        \item \textbf{Operandi non definiti}: ad esempio in alcuni linguaggi è
          possibile scrivere espressioni non definite, ad esempio:
          \begin{center}
            \texttt{a == 0 ? b : b / a}
          \end{center}
          è un'espressione non definita se \( a = 0 \)
          perchè si avrebbe una divisione per zero. Ci sono due tipi di valutazioni:
          \begin{itemize}
            \item \textbf{Valutazione eager}: si valutano prima tutte le parti
              dell'espressione

            \item \textbf{Valutazione lazy}: si valutano solo le parti necessarie
              per applicare l'operatore, quindi in questo caso si valuterebbe solo
              \( a \) e soltanto se \( a \neq 0 \) si valuterebbe \( b / a \).
          \end{itemize}

        \item \textbf{Ordine dei side effects}: Consideriamo il seguente codice:
          \[
            \begin{aligned}
                &\texttt{a := 10;} \\
                &\texttt{b := a + fun(\&a);} \\
            \end{aligned}
          \] 
          dove \( fun \) è una funzione che prende un puntatore ad una variabile
          e modifica il suo parametro. Supponiamo che \texttt{fun(\&a)} imposti
          il valore di \texttt{a} a 12 e lo ritorni.
          \begin{itemize}
            \item Se si valuta prima \texttt{a} e poi si esegue \texttt{fun(\&a)},
            si avrebbe \texttt{b = 10 + fun(\&a)} e dopo l'esecuzione di
            \texttt{fun(\&a)} si avrebbe \texttt{a = 12} e
            \[
              \texttt{b = 10 + 12 = 22}
            \] 

          \item Se si valuta prima \texttt{fun(\&a)} e poi \texttt{a}, si avrebbe
            \texttt{b = a + fun(\&a)} e dopo l'esecuzione di \texttt{fun(\&a)}
            si avrebbe \texttt{a = 12} e
            \[
              \texttt{b = 12 + 12 = 24}
            \]
          \end{itemize}
          Quindi l'ordine di valutazione degli operandi può influenzare il
          risultato di un'espressione se ci sono side effects.
      \end{itemize}

    \item \textbf{Presenza di side effects}: cioè se la valutazione di un'espressione
      modifica lo stato/memoria

    \item \textbf{Overloading degli operatori}: cioè se un operatore può essere
      applicato a più tipi di operandi, ad esempio in alcuni linguaggi il \( + \)
      può essere applicato a interi come somma, oppure a stringhe come concatenazione

    \item \textbf{Espressioni con operandi di tipo misto}: cioè se un operatore può
      essere applicato a operandi di tipi diversi, ad esempio in alcuni linguaggi
      è possibile sommare un intero ad una stringa applicando casting automatici
\end{itemize}

\subsubsection{Sintassi}
Definiamo la sintassi del linguaggio IMP per le espressioni:
\[
  \begin{aligned}
    \texttt{<Expr} \quad &\texttt{E} \to \texttt{A} \mid \texttt{B} \\
                         &\texttt{A} \to \texttt{N} \mid  \texttt{A op A} \\
                         &\texttt{B} \to \texttt{true} \mid \texttt{false} \\
                         &\mid \texttt{A = A} \mid \texttt{not B} \mid \texttt{B or B}
  \end{aligned}
\] 
Dove:
\begin{itemize}
  \item \( N \) sono i numeri naturali, considerati parte del vocabolario
  \item \( op \) è l'insieme di operatori aritmetici \( \{+, -, *\} \)
  \item \texttt{A} sono le espressioni aritmetiche
  \item \texttt{B} sono le espressioni booleane
\end{itemize}

\subsubsection{Semantica per espressioni aritmetiche}
Consideriamo:
\begin{itemize}
  \item \( \texttt{N} \): l'insieme dei valori numerici \( n, m, p \) 
  \item \( \mathcal{E} \): l'insieme di espressioni valutabili, ovvero alberi di
    valutazione di espressioni \( e \). Cioè una stringa di soli terminali generati
    in Exp
\end{itemize}

\vspace{1em}
\noindent
Diamo una semantica per le \textbf{espressioni aritmetiche} definendo un sistema di transizione:
\[
  Stati: \mathcal{E} \cup \texttt{N} \; \text{(stati terminali)}
\] 
Le \textbf{regole di transizione} sono definite attraverso regole costruite
induttivamente sulla grammatica:
\[
  op \in \left\{ +, -, * \right\}
\] 
Le regole di transizione per le espressioni sono:
\begin{itemize}
  \item Regola 1 (Assioma): Definisce \( p \) come la semantica del calcolo
    effettivo di un operatore
    \[
      \mathcal{E}_1: \overbrace{\texttt{m op n}}^{\text{simboli}} \to p
      = \underbrace{\texttt{m op n}}_{\text{significato}}
    \] 

  \item Regola 2: Definisce la valutazione dell'espressione a sinistra per prima
    \[ \mathcal{E}_2:\; 
      \begin{prooftree}
        \hypo{e_0 \to e'_0}
        \infer1{e_0 \texttt{ op } e_1 \to e'_0 \texttt{ op } e_1}
      \end{prooftree}
    \]

  \item Regola 3: Definisce la valutazione dell'espressione a destra con un'espressione
    già valutata a sinistra
    \[ \mathcal{E}_3:\;
      \begin{prooftree}
        \hypo{e_1 \to e'_1}
        \infer1{m \texttt{ op } e_1 \to m \texttt{ op } e'_1}
      \end{prooftree}
    \] 
\end{itemize}
Le regole \( \mathcal{E}_2 \) e \( \mathcal{E}_3 \) impongono un ordine di valutazione
delle espressioni, in questo caso da sinistra a destra.
\begin{example}
  Consideriamo la seguente espressione valutandola seguendo le regole appena definite:
  \[
    \underbrace{(2+3)}_{e_0} * \underbrace{(5 - (1+4))}_{e_1}
  \] 
  Si ha il seguente albero di valutazione:
  \begin{center}
    \begin{forest}
      [*
        [+, Blue
          [2, Blue]
          [3, Blue]
        ]
          [-, Red
          [5, Red]
          [+, Orange
            [1, Orange]
            [4, Orange]
          ]
        ]
      ]
    \end{forest}
  \end{center}
  Le regole che abbiamo fissato non ci permettodno di valutare prima \( e_1 \) e poi
  \( e_0 \), ma ci obbliga a valutare prima \( e_0 \) e poi \( e_1 \). Si potrebbe
  aggiungere la seguente regola alternativa per permettere di valutare prima \( e_1 \):
  \[ 
    \mathcal{E}_3':\;
    \begin{prooftree}
      \hypo{e_1 \to e'_1}
      \infer1{e_0 \texttt{ op } e_1 \to e_0 \texttt{ op } e'_1}
    \end{prooftree}
  \] 
\end{example}

\subsubsection{Semantica per espressioni booleane}
Diamo una semantica per le \textbf{espressioni booleane}:
Gli stati terminali sono i seguenti:
\[
  \texttt{B} = \left\{ \texttt{true}, \texttt{false} \right\}
\] 
Definiamo le regole di transizione continuando da quelle già definite per le
espressioni aritmetiche:
\begin{itemize}
  \item Regola 4:
    \[
    \mathcal{E}_4: \; \underbrace{t_1 \texttt{ or } t_2}_\text{simbolo} \to t
    = \underbrace{t_1 \texttt{ or } t_2}_\text{significato}
    \quad t, t_1, t_2 \in \texttt{B}
    \] 

  \item Regola 2: Vale anche per le espressioni booleane

  \item Regola 3: Adattata per le espressioni booleane
    \[
      \mathcal{E}_3':\;
      \begin{prooftree}
        \hypo{e_1 \to e'_1}
        \infer1{t \texttt{ or } e_1 \to t \texttt{ or } e_1'}
      \end{prooftree}
    \] 

  \item Uniamo \( \mathcal{E}_3 \) con \( \mathcal{E}_3' \)
    \[
      \begin{prooftree}
        \hypo{e_1 \to e'_1}
        \infer1{k \texttt{ bop } e_1 \to k \texttt{ bop } e_1'}
      \end{prooftree}
    \] 
    con \( k \in \texttt{N} \cup \texttt{B} \) e \( bop \in \{or, +, -, *\} \)

  \item Regola 5:
    \[
      \mathcal{E}: \;
      \begin{prooftree}
        \hypo{e \to e'}
        \infer1{\texttt{not } e \to \texttt{not } e'}
      \end{prooftree}
    \] 

  \item Regola 6:
    \[
      \mathcal{E}_6: \; \texttt{not } e_t \to t' = \texttt{not } t \quad t, t' \in \texttt{B}
    \] 
\end{itemize}
Il sistema di transizione che abbiamo definito è:
\( \left<\mathcal{E} \cup \underbrace{\texttt{N} \cup \texttt{B} }_{\text{Terminali}} \right>\) 

\begin{definition}[Valutazione]
  La funzione di valutazione:
  \[
    Eval:\; Exp \to \overbrace{\texttt{N} \cup \texttt{B}}^{\text{Con}}
  \] 
  descrive il comportamento dinamico delle espressioni:
  \[
    \forall e \in \mathcal{E} \;.\; Eval(e) = k \iff e \to^*_e k
  \] 
  dove \( \to^*_e \) è la chiusura transitiva di \( \to  \), cioè:
  \[
    \exists n \;.\; e \to e_1 \to e_2 \to \ldots \to e_n = k
  \] 
  con \( n \) transizioni.
\end{definition}

\begin{definition}[Equivalenza di espressioni]
  Definiamo la relazione di equivalenza \( \equiv \subseteq Exp \times Exp \) tra
  espressioni come:
  \[
    e_1 \equiv e_2 \iff Eval(e_1) = Eval(e_2)
  \] 
  cioè due espressioni sono equivalenti se vengono valutate allo stesso valore.
\end{definition}

\subsection{Identificatori}
Gli identificatori sono \textbf{sequenze di caratteri} usate per riferirsi ad
\textbf{oggetti denotabili}, ovvero a variabili, costanti, funzioni, procedure, ecc...

\begin{example}
  Alcuni esempi di identificatori sono i seguenti:
  \[
  \begin{aligned}
    &\texttt{const \textcolor{Blue}{pi} = 3.14;} \\
    &\texttt{int \textcolor{Blue}{x};} \\
    &\texttt{void \textcolor{Blue}{f}(...)\{...\}}
  \end{aligned}
  \] 
  Ogni identificatore crea un legame tra il nome dell'identificatore e l'oggetto
  che denota, ad esempio \texttt{pi} denota la costante \( 3.14 \), \texttt{x} denota
  una variabile di tipo intero e \texttt{f} denota una procedura.
\end{example}

\begin{definition}[Oggetti denotabili]
  Sono oggetti del linguaggio di programmazione a cui si può fare riferimento mediante
  l'uso di identificatori
\end{definition}

\noindent
 Nel nostro linguaggio gli 
identificatori sono definiti nel seguente insieme:
\[
  Id = \left< \texttt{a}, \texttt{rate}, \texttt{b20}, \ldots, \texttt{id}, \texttt{x}, \ldots \right>
\] 

\subsubsection{Aspetti implementativi}
Gli aspetti implementativi da tenere in considerazione nella definizione degli
identificatori sono i seguenti:
\begin{itemize}
  \item \textbf{Nomi case sensitive o case insensitive}
  \item \textbf{Parole riservate}
  \item \textbf{Vincoli sulla lunghezza}
  \item \textbf{Presenza di caratteri speciali}
\end{itemize}
Le parole chiave del nostro linguaggio oppure altre specifiche che dipendono dal
linguaggio non si possono usare come identificatori.

\subsubsection{Legami, scope e occorrenze}
\begin{definition}[Legami (binding)]
  Consideriamo il seguente esempio:
  \[
    \left( 
      \sum_{\mathcolor{Green}{i=1}}^{\mathcolor{Red}{n}}
      \left( 
        \sum_{\mathcolor{Green}{j=1}}^{\mathcolor{Red}{m}} a_{\color{Blue}ij} \ldots b_{\color{Blue}i\color{Red}k}
      \right) 
    \right) 
  \] 
  \begin{itemize}
    \item \color{Green}Le \textbf{definizioni} creano dei legami tra il nome e
      il significato, cioè crea \textbf{occorrenze di binding}.\color{black}

    \item \color{Blue}Gli usi del legame, cioè \textbf{occorrenze d'uso (applied)}
      sono quelli che utilizzano il nome per far riferimento al significato\color{black}

    \item \color{Red}Le \textbf{occorrenze libere} sono quelle che non fanno riferimento
      ad alcun legame, quindi il significato non è definito.\color{black}
  \end{itemize}
\end{definition}

\begin{definition}[Scope]
  Lo scope è il \textbf{raggio di azione} di una definizione. Determina lo spazio in
  cui le potenziali occorrenze dell'identificatore definito fanno riferimento a
  quella definizione \( def \).
  \[
    \color{Green}\underbrace{\color{black}
    \left( 
      \sum_{\mathcolor{Green}{\underbrace{i=1}_{def}}}^{\mathcolor{Red}{n}}
      \color{Blue}\underbrace{\color{black}
      \left( 
        \sum_{\mathcolor{Blue}{\underbrace{j=1}_{def}}}^{\mathcolor{Red}{m}} \overbrace{a_{\color{Blue}ij} \ldots b_{\color{Blue}i\color{Red}k\color{black}}}^{\color{Blue}\text{uso}}
      \right)}_{\color{Blue}\text{scope } j\color{black}}
    \right)}_{\color{Green}\text{scope } i\color{black}}
  \] 
\end{definition}

\vspace{1em}
\noindent
Nei linguaggi di programmazione gli identificatori possono essere in diverse posizioni
e sono le seguenti:
\begin{itemize}
  \item \textbf{Definizione} (occorrenze di binding): Un identificatore è in posizione di definizione
    quando si attribuisce il significato dell'identificatore

  \item \textbf{Uso} (occorrenze di uso): Un identificatore è in posizione di
    uso quando l'occorrenza serve per riferirsi al significato determinato da una
    definizione. Deve trovarsi nello scope (raggio di azione) di una definizione.

  \item \textbf{Libera}: Un identificatore è in posizione libera se laa sua
    occorrenza non si trova nello scope (raggio di azione) di una definizione.
\end{itemize}

\subsubsection{Tipi di binding}
Nel seguente esempio i legami tra i nomi e i significati sono:
\[
  \begin{aligned}
    &\texttt{const pi = 3.14;} \quad& \text{costante} \\
    &\texttt{int x;} \quad& \text{locazione in memoria} \\
    &\texttt{void f(...)\{...\}} \quad& \text{pezzo di codice}
  \end{aligned}
\] 
Esistono due tipi di binding:
\begin{itemize}
  \item \textbf{Binding statici}:
    Sono legami che non cambiano durante l'esecuzione del
    programma, a meno che non vengano sovrascritti da un altro legame. Un legame
    può essere:
    \begin{itemize}
      \item Creato
      \item Distrutto
      \item Sovrascritto
      \item \textbf{Non può essere modificato}
    \end{itemize}

  \item \textbf{Binding dinamici}:
    Alcuni legami possono cambiare, ad esempio il legame tra una locazione e un valore.
    Questi legami sono \textbf{dinamici}, cioè uno degli elementi del legame cambia
    durante l'esecuzione del programma. Ad esempio il valore di una variabile può
    cambiare, ma il nome della variabile e la locazione in memoria a cui fa riferimento
    non cambiano.
\end{itemize}

\begin{definition}[Tempo di binding]
  È il momento in cui un legame viene creato e dipende dal linguaggio e anche dagli
  oggetti. Ad esempio le costanti presenti nel linguaggio vengono legate a tempo
  di progettazione, mentre le costanti definite dall'utente vengono legate a tempo di
  implementazione, cioè quando si compila il programma.
\end{definition}

\subsubsection{Ambiente}
I legami vengono inseriti nell'\textbf{ambiente}, che non è altro che un
\textbf{insieme di binding}. In un programma tutti gli identificatori devono essere
definiti, cioè non possono mai essere in \textbf{posizione libera}.

Ci sono due tipi di termini:
\begin{itemize}
  \item \textbf{Termini ground}:
    Un termine è detto \textbf{ground} se non ha identificatori e ha sempre significato.
    Nel nostro caso le espressioni che abbiamo definito sono ground.

  \item \textbf{Tremini chiusi}:
    Un termine è detto \textbf{chiuso} se non ha identificatori in posizione libera
\end{itemize}
Entrambi questi tipi di termini hanno significato senza dover ricorrere ad un
contesto esterno, cioè ad un ambiente.

\subsubsection{Aggiungiamo gli identificatori alla sintassi delle espressioni}
Riprendiamo la seguente notazione:
\begin{itemize}
  \item \( \mathcal{E} \): l'insieme di espressioni

  \item \( DVal = \texttt{N} \cup \texttt{B} \): l'insieme di valori numerici e
    booleani, che sono anche \textbf{valori denotabili}
\end{itemize}

\vspace{1em}
\noindent
Definiamo un ambiente per le espressioni:
\[
  \begin{aligned}
    &Env = \bigcup_{I \subseteq_f Id} Env_I \\
    &Env_I:\; I \to Dval \quad \rho \text{ metavariabile per un ambiente specifico}
  \end{aligned}
\] 
L'ambiente è una funzione che associa un valore denotabile ad ogni identificatore
in un insieme \( I \) finito (\( \subseteq_f \) sottoinsieme finito).

\begin{example}
  Consideriamo i seguenti identificatori:
  \[
    I = \left\{ x, y \right\}
  \] 
  Allora un esempio di ambiente è il seguente:
  \[
  \begin{aligned}
    \rho \in Env_I: \quad &\rho(x) = 5 \\
                 &\rho(y) = 5
  \end{aligned}
  \] 
  L'espressione \( x + y - 5 \) ha significato soltanto se valutata nell'ambiente
  \( \rho \):
  \[
    \rho \vdash x + y - 5
  \] 
\end{example}

\vspace{1em}
\noindent
Ora aggiungiamo alla sintassi delle espressioni gli identificatori 
con \( \mathcolor{Blue}{x} \in Id \):
\[
  \begin{aligned}
    \texttt{<Expr} \quad &\texttt{E} \to \texttt{A} \mid \texttt{B} \\
                         &\texttt{A} \to \texttt{N}  \mid \mathcolor{Blue}{x} \mid  \texttt{A op A} \\
                         &\texttt{B} \to \texttt{true} \mid \texttt{false}  \mid \textcolor{Blue}{x} \\
                         &\mid \texttt{A = A} \mid \texttt{not B} \mid \texttt{B or B}
  \end{aligned}
\] 


\subsubsection{Semantica per espressioni con identificatori}
Definiamo la semantica per le espressioni da valutare in un ambiente \( \rho  \):
\[
  \rho \vdash e \to e'
\] 
questo significa che l'espressione \( e \) viene valutata in \( e' \) nell'ambiente \( \rho \).
Le regole di transizione per le espressioni con identificatori diventano:
\begin{itemize}
  \item Regola 1 (Assioma):
    \[
      \rho \vdash \texttt{m op n} \to p = \texttt{m op n}
    \] 

  \item Regola 2:
    \[
      \begin{prooftree}
        \hypo{\rho \vdash e_0 \to e'_0}
        \infer1{\rho \vdash e_0 \texttt{ bop } e_1 \to e'_0 \texttt{ bop } e_1}
      \end{prooftree}
    \]
    con \( bop \in \{\texttt{or}, +, -, *\} \)

  \item Regola 3+4:
    \[
      \begin{prooftree}
        \hypo{\rho \vdash e_1 \to e'_1}
        \infer1{\rho \vdash k \texttt{ bop } e_1 \to k \texttt{ bop } e'_1}
      \end{prooftree}
    \]
    con \( bop \in \{\texttt{or}, +, -, *\} \)

  \item Regola 5:
    \[
      \begin{prooftree}
        \hypo{\rho \vdash e \to e'}
        \infer1{\rho \vdash \texttt{not } e \to \texttt{not } e'}
      \end{prooftree}
    \]

  \item Regola 6:
    \[
      \rho \vdash \texttt{not } t \to t' = \texttt{not } t
    \]

  \item Regola 7:
    \[
      \rho \vdash x \to k \text{ se } \rho(x) = k
    \]
\end{itemize}
Sono le stesse regole di transizione che avevamo definito per le espressioni senza
identificatori, ma ora sono valutate in un ambiente \( \rho \) che viene propagato in
tutte le transizioni.

Ora definiamo una nuova regola che utilizza l'ambiente \( \rho \) per valutare un
identificatore \( x \):
\begin{itemize}
  \item Regola 7:
    \[
      \rho \vdash x \to k \text{ se } \rho(x) = k
    \]
\end{itemize}

\begin{example}
  Consideriamo la seguente espressione:
  \[
    \texttt{x + y - 5}
  \] 
  da valutare nell'ambiente:
  \[
    \rho = [\texttt{x} \mapsto 2, \texttt{y} \mapsto 5]
  \] 

  \vspace{1em}
  \noindent
  Applichiamo le regole:
  \begin{enumerate}
    \item \[
        \begin{aligned}
          &\begin{prooftree}
            \hypo{\rho \vdash \texttt{x} \to 2}
            \infer1[$\rho(x) = 2$]{\rho \vdash \texttt{x + y - 5} \to 2 + \texttt{y - 5}}
          \end{prooftree}\\
          &\qquad\qquad\qquad \downarrow\\
          &\begin{prooftree}
            \hypo{\rho \vdash \texttt{y} \to 5}
            \infer1[$\rho(y)=5$]{\rho \vdash \texttt{y - 5} \to 5 - 5}
            \infer1{\rho \vdash \texttt{2 + y - 5} \to 2 + 5 - 5}
          \end{prooftree}\\
          &\qquad\qquad\qquad \downarrow\\
          &\quad \begin{prooftree}
            \hypo{\rho \vdash 5-5 \to 0}
            \infer1{\rho \vdash 2 + 5 - 5 \to 2 + 0}
            \infer1{\rho \vdash 2 + 0 \to 2}
          \end{prooftree}
        \end{aligned}
    \] 
  \end{enumerate}
\end{example}

\subsection{Tipo di dato}
Il tipo è l'insieme dei valori denotabili da un identificatore e l'insieme degli
operatori applicabili, tali valori devono
condividere una certa proprietà. Ad esempio, il tipo \texttt{int} è l'insieme dei
numeri interi, il tipo \texttt{bool} è l'insieme dei valori booleani, ecc...

L'utilizzo del tipo ha varie funzionalità:
\begin{itemize}
  \item \textbf{A tempo di progettazione}: permette di organizzare le funzionalità
    del linguaggio.

  \item \textbf{A tempo di sviluppo}: servono a prevenire errori di programmazione,
    ad esempio se si cerca di sommare una stringa ad un intero, il compilatore può
    segnalare un errore di tipo.

  \item \textbf{A tempo di implementazione}: Servono per capire quanta memoria
    allocare per una certa variabile.
\end{itemize}

\vspace{1em}
\noindent
Il tipo deve essere legato alla variable, quindi viene aggiunto al legame con l'oggetto
denotato. Si dovrà creare un ambiente separato per i tipi e si avrà:
\begin{itemize}
  \item \textbf{Semantica statica}, che viene utilizzata durante la compilazione per
    verificare la correttezza dei tipi e per allocare la memoria necessaria per
    le variabili. \textbf{Verifica che le espressioni siano scritte correttamente}.

  \item \textbf{Semantica dinamica}, come quella definita per le espressioni
    (\( \rho  \)), che viene utilizzata durante l'esecuzione. Infatti la valutazione
    delle espressioni \textbf{assume che siano scritte correttamente}.
\end{itemize}

\subsubsection{Ambiente statico}
È l'insieme di legami tra identificatori e tipi. I tipi denotabili sono definiti
dal seguente insieme:
\[
  DTyp = \left\{ \texttt{int}, \texttt{bool} \right\}
\] 
Con valori denotabili \( DVal \) che sono i valori numerici e booleani, quindi:
\[
  DVal = \texttt{N} \cup \texttt{B}
\] 
L'insieme di tutti gli ambienti è l'insieme di tutti i possibili ambienti
di un certo identificatore \( I \):
\[
  TEnv = \bigcup_{I \subseteq_f Id} TEnv_I
\] 
dove per ogni identificatore in \( I \) associa un tipo in \( DTyp \) 
\[
  TEnv_I:\; I \to DTyp
\] 

\vspace{1em}
\noindent
La metavariabile utilizzata per rappresentare un generico ambiente statico è
\( \Delta \in TEnv \). La seguente notazione asserisce che l'espressione \( e \) 
è di tipo \( \tau \in DTyp \) nell'ambiente statico \( \Delta \) definito sugli
identificatori in \( I \):
\[
  \Delta \vdash_I e : \tau
\] 
Quando l'ambiente non è indicato, si sottintende l'insieme vuoto:
\[
  \vdash e : \tau \equiv \varnothing \vdash e : \tau
\] 
Cioè per ogni ambiente è possibile applicare la regola di semantica statica.

Se le regole della semantica statica permettono di associare un tipo ad un'espressione,
allora l'espressione è \textbf{ben formata}

\subsubsection{Semantica statica per le espressioni}
La semantica statica per le espressioni associa un tipo alle espressioni ben formate,
quindi verifica solo vincoli di tipo tra operandi.
Definiamo la semantica statica per le espressioni:
\begin{itemize}
  \item Regola 1: con \( n \in \texttt{N} \)
    \[
      \mathcal{E}_{S_1}: \; \vdash n : \texttt{int}
    \] 

  \item Regola 2: con \( t \in \texttt{B} \)
    \[
      \mathcal{E}_{S_2}: \; \vdash t : \texttt{bool}
    \] 

  \item Regola 3:
    \[
      \mathcal{E}_{S_3}: \;
      \Delta \vdash_I x : \tau \quad\text{ se }\quad x \in I,\; \Delta(x) = \tau
    \] 

  \item Regola 4:
    \[
      \mathcal{E}_{S_4}: \;
      \begin{prooftree}
        \hypo{\Delta \vdash_I e_1 : \texttt{int}}
        \hypo{\Delta \vdash_I e_2 : \texttt{int}}
        \infer2{\Delta \vdash_I e_1 \texttt{ op } e_2 : \texttt{int}}
      \end{prooftree}
    \] 

  \item Regola 5:
    \[
      \mathcal{E}_{S_5}: \;
      \begin{prooftree}
        \hypo{\Delta \vdash_I e_1 : \texttt{bool}}
        \hypo{\Delta \vdash_I e_2 : \texttt{bool}}
        \infer2{\Delta \vdash_I e_1 \texttt{ or } e_2 : \texttt{bool}}
      \end{prooftree}
    \]

  \item Regola 6:
    \[
      \mathcal{E}_{S_6}: \;
      \begin{prooftree}
        \hypo{\Delta \vdash_I e : \texttt{bool}}
        \infer1{\Delta \vdash_I \texttt{not } e : \texttt{bool}}
      \end{prooftree}
    \]

  \item Regola 7:
    \[
      \mathcal{E}_{S_7}: \;
      \begin{prooftree}
        \hypo{\Delta \vdash_I e_1 : \texttt{int}}
        \hypo{\Delta \vdash_I e_2 : \texttt{int}}
        \infer2{\Delta \vdash_I e_1 = e_2 : \texttt{bool}}
      \end{prooftree}
    \]
\end{itemize}

\subsubsection{Ambienti compatibili}
\begin{definition}
  Considerando \( I \subseteq_f Id \) 
  \begin{itemize}
    \item Ambiente dinamico \( \rho \in Env_I \) che associa valori a nomi
    \item Ambiente statico \( \Delta \in TEnv_I \) che associa tipi a identificatori
  \end{itemize}
  Si dice che \( \rho  \) e \( \Delta \) sono compatibili se
  \[
    \forall id \in I \;.\; \Delta(id) = \tau \iff \rho(id) \in \tau
  \] 
  cioè se per ogni identificatore \( id \) in \( I \), il tipo associato a \( id \) in
  \( \Delta \) è \( \tau \) se e solo se il valore associato a \( id \) in \( \rho \) è
  un elemento di \( \tau \).

  Per dire che due ambienti \( \rho  \) e \( \Delta \) sono compatibili, si scrive:
  \[
    \rho \vdash_{\Delta}
  \] 
\end{definition}

\begin{example}
  Consideriamo ad esempio i seguenti ambienti:
  \begin{itemize}
    \item Ambienti compatibili
      \[
        \begin{aligned}
        &\Delta = \left[ x \mapsto \texttt{int}, y \mapsto \texttt{bool} \right] \\
        &\rho = \left[ x \mapsto 2, y \mapsto \texttt{true} \right]
        \end{aligned}
      \] 

    \item Ambienti non compatibili
      \[
        \begin{aligned}
        &\Delta = \left[ x \mapsto \texttt{int}, y \mapsto \texttt{bool} \right] \\
        &\rho = \left[ x \mapsto 2, y \mapsto 3 \right]
        \end{aligned}
      \] 
  \end{itemize}
\end{example}

\noindent
Per definire gli ambienti (sia \( \Delta \) che \( \rho  \)) c'è bisogno di una
categoria sintattica che permette di creare e modificare legami che vanno a costituire
gli ambienti

\subsection{Dichiarazioni}
La dichiaraazione è una \textbf{categoria sintattica} che fornisce il meccanismo (implicito
o esplicito) per \textbf{creare legami} tra identificatori e tipi (statico) o oggetti
(dinamico). Le dichiarazioni denotano richieste di 
\textbf{modifica o creazione di ambienti}.

\vspace{1em}
\noindent
Le dichiarazioni vengono \textbf{elaborate} per ottenere le richieste di modifica o
creazione di ambienti

\subsubsection{Sintassi delle dichiarazioni}
Definiamo la sintassi delle dichiarazioni:
\[
  \begin{aligned}
    \mathcal{D}: \; D & \to  \texttt{ nil}  \\
                      & \mid \texttt{const x: } \tau = e  \\
                      & \mid D; D  \mid D \texttt{ in } D \\
                      & \mid \rho 
  \end{aligned}
\] 
dove:
\begin{itemize}
  \item \texttt{nil}: corrisponde a \( \Delta = \varnothing \) 

  \item \( \texttt{const x: } \tau = e \):  è la dichiarazione di un \( id \)  costante
    di nome \( x \), tipo \( \tau \) e valore rappresentato da \( e \).

  \item \( D; D \): è la composizione sequenziale, cioè prima si elabora la
    dichiarazione a sinistra e poi quella a destra

  \item \( D \texttt{ in } D \): è la composizione privata, cioè i legami della
    dichiarazione a sinistra sono visibili solo nella dichiarazione a destra

  \item \( \rho \): è un ambiente dinamico presente nella grammatica per motivi formali
\end{itemize}

\subsubsection{Aggiornamento di un ambiente}
Consideriamo gli ambienti \( \beta \) e \( \beta' \). La modifica di \( \beta \) da
parte di \( \beta' \) si scrive:
\[
  \beta \left[ \beta' \right]
\] 
Consideriamo che \( \beta \) sia definito sugli identificatori \( I \):
\[
  \beta:\; I \quad I \subseteq_f Id
\] 
e \( \beta' \) sia definito sugli identificatori \( I' \):
\[
  \beta':\; I' \quad I' \subseteq_f Id
\] 
Il nuovo ambiente parlerà sia degli identificatori in \( I \) che di quelli in \( I' \):
\[
  \beta \left[ \beta' \right] = \overline{\beta} \quad \overline{\beta}:\; I \cup I'
\] 
Se l'identificatore è presente sia nell'ambiente \( \beta \) che in \( \beta' \), allora
la priorità è data a \( \beta' \):
\[
  \forall x \in I' \;.\; \overline{\beta}(x) = \beta'(x)
\] 
Se l'identificatore è presente solo in \( \beta \) allora si mantiene lo stesso legame:
\[
  \forall x \in I \setminus I' \;.\; \overline{\beta}(x) = \beta(x)
\] 
Quindi \( \overline{\beta} \) è un nuovo ambiente in cui si prendono tutti i legami
di \( \beta \) e \( \beta' \) e per quelli in comune si tengono solo quelli di
\( \beta' \). Di fatto \( \beta' \) \textbf{sovrascrive} \( \beta \).

\vspace{2em}
\noindent
\begin{example}
  Consideriamo i seguenti ambienti:
  \[
    \begin{aligned}
      &\rho = \left[ x \mapsto 2, y \mapsto \texttt{true}, z \mapsto 5 \right] \\
      &\rho' = \left[ x \mapsto 5, w \mapsto \texttt{false} \right]
    \end{aligned}
  \] 
  Allora l'aggiornamento di \( \rho \) da parte di \( \rho' \) è il seguente:
  \[
    \rho \left[ \rho' \right] =
    \left[ x \mapsto 5, w \mapsto \texttt{false}, y \mapsto \texttt{true}, z \mapsto 5 \right]
  \]
  Si vede che \( x \) è stato sovrascritto da \( \rho' \), mentre \( y \) e \( z \)
  sono rimasti invariati e \( w \) è stato aggiunto.
\end{example}

\subsubsection{Composizione sequenziale}

Ogni dichiarazione successiva o aggiunge nuovi legami o sovrascrive legami esistenti.
L'ambiente risultante è così visibile al codice che viene successivamente

\begin{example}
  Consideriamo ad esempio il seguente codice:
\begin{lstlisting}
const x: int = 3;
const x: int = 5; const y: int = 6 * x;
const z: int = x + y;
\end{lstlisting}
  Osserviamo i valori di questo codice:
  \begin{enumerate}
    \item A riga 1 il valore assegnato a \( x \) è \( \rho = [x \mapsto 3] \) 
    \item A riga 2 il valore assegnato a \( x \) è \( \rho[x \mapsto 5] = [x \mapsto 5] = \rho_1 \) e
      viene usata per elaborare \( y \) che assume il valore
      \( \rho_2 = [x \mapsto 5, y \mapsto 6 * 5 = 30] \)
    \item A riga 3 viene usato il valore di \( x \) dichiarato a riga 2 e il valore
      di \( y \) dichiarato a riga 2, cioè \( (\rho[\rho_1])[\rho_2] = [x \mapsto 5, y \mapsto 30] \),
      per elaborare \( z \) che assume il valore \( \rho_3 = [x \mapsto 5, y \mapsto 30, z \mapsto 5 + 30 = 35] \)
  \end{enumerate}
\end{example}

\subsubsection{Composizione privata}

I legami creati o sovrascritti da \( D_1 \) sono \textbf{ad uso esclusivo}
dell'elaborazione di \( D_2 \) e \textbf{non visibili} al codice successivo.

\begin{example}
  Consideriamo ad esempio il seguente codice:
\begin{lstlisting}
const x: int = 3;
const x: int = 5 in y: int = 6 * x;
const z: int = x + y;
\end{lstlisting}
  Osserviamo i valori di questo codice:
  \begin{enumerate}
    \item A riga 1 il valore assegnato a \( x \) è \( \rho = [x \mapsto 3] \) 
    \item A riga 2 il valore assegnato a \( x \) è \( \rho_1 = [x \mapsto 5] \) e
      viene usata per elaborare \( y \) che assume il valore
      \( \rho_2 = [y \mapsto 6 * 5 = 30] \)
    \item A riga 3 viene usato il valore di \( x \) dichiarato a riga 1 e il valore
      di \( y \) dichiarato a riga 2, cioè \( \rho[\rho_2] = [x \mapsto 3, y \mapsto 30] \),
      per elaborare \( z \) che assume il valore \( \rho_3 = [x \mapsto 3, y \mapsto 30, z \mapsto 3 + 30 = 33] \)
  \end{enumerate}
\end{example}

\subsubsection{Identificatori in posizione libera e in posizione di definizione}
Caratterizziamo le dichiarazioni:
\begin{itemize}
  \item \textbf{In posizione libera (FI)}: Associa ad ogni costrutto sintattico
    l'insieme degli identificatori \textbf{liberi}, cioè identificatori che non hanno
    un legame e richiedono un contesto per avere significato.
    \[
      FI(Exp \cup Dec) = I \subseteq Id \quad FI:\; Exp \cup Dec \to \mathcal{P}(Id)
    \] 
    Definiamo \( FI \) per induzione:
    \begin{itemize}
      \item \textbf{Caso base}: \( FI(k) = \varnothing \) con \( k \in \texttt{N} \cup \texttt{B} \)
      \item \textbf{Passo induttivo}:
        \begin{itemize}
          \item Espressioni:
            \begin{itemize}
              \item \( FI(x) = \left\{ x \right\} \) 
              \item \( FI(e_1 \texttt{ op } e_2) = FI(e_1) \cup FI(e_2) = FI(e_1 \texttt{ or } e_2) \) 
              \item \( FI(\texttt{not } e) = FI(e)\) 
            \end{itemize}

          \item Dichiarazioni:
            \begin{itemize}
              \item \( FI(\texttt{nil}) = \varnothing \) 
              \item \( FI(\texttt{const x: } \tau = e) = FI(e) \) 
              \item \( FI(d_1; d_2) = FI(d_1) \cup (FI(d_2) \setminus DI(d_1) = FI(d_1 \texttt{ in } d_2) \) dove
                \( DI(d_1) \) è l'insieme degli identificatori definiti da \( d_1 \) 
              \item \( FI(\rho) = \varnothing \) 
            \end{itemize}
        \end{itemize}
    \end{itemize}
    Questa funzione fornisce l'insieme degli identificatori per cui bisogna cercare
    un significato nell'ambiente per poter valutare l'espressione o elaborare la
    dichiarazione.

  \item \textbf{In posizione di definizione (DI)}: Restituisce gli identificatori
    che nel costrutto vengono legati ad un significato che è visibile all'esterno
    del costrutto
    \[
      DI: \; Exp \cup Dec \to \mathcal{P}(Id)
    \]
    Definiamo \( DI \):
    \begin{itemize}
      \item Espressioni: Le espressioni non definiscono identificatori, quindi:
        \[
          \forall e \in Exp \;.\; DI(e) = \varnothing
        \] 
      \item Dichiarazioni:
        \begin{itemize}
          \item \( DI(\texttt{nil}) = \varnothing \) 
          \item \( DI(\texttt{const x: } \tau = e) = \left\{ x \right\} \) 
          \item \( DI(\rho) = I \quad \rho : I \; (\rho \in Env_I)\)
          \item \( DI(d_1 \texttt{ in } d_2) = DI(d_2) \): solo gli identificatori
            legati in \( d_2 \) hanno significato \textbf{dopo} l'elaborazione della
            composizione
          \item \( DI(d_1; d_2) = DI(d_1) \cup DI(d_2) \): tutti gli identificatori
            legati in \( d_1 \) e \( d_2 \) restano legati al loro significato dopo
            l'elaborazione della composizione
        \end{itemize}
    \end{itemize}
\end{itemize}

\subsubsection{Semantica statica per le dichiarazioni}
La semantica statica per le dichiarazioni associa un ambiente statico \( \Delta \) 
alle dichiarazioni ben formate.
Definiamo la semantica statica per le dichiarazioni:
\[
  \Delta \vdash d: \Delta'
\] 
dove:
\begin{itemize}
  \item \( \Delta \) è il contesto, cioè eventuali dichiarazioni precedenti
  \item \( \Delta' \) è un ambiente definito (e quindi associato) da \( d \)
\end{itemize}
Le regole per le dichiarazioni sono:
\begin{itemize}
  \item Regola 1:
    \[
      \mathcal{D}_{S_1}: \; \vdash \texttt{nil} : \varnothing
    \] 

  \item Regola 2:
    \[
      \mathcal{D}_{S_2}: \; \vdash \rho : \Delta
    \]
    dove \( \rho  \) e \( \Delta \) sono compatibili.
    \begin{example}
      Ad esempio:
      \[
        \rho = \left[ x \mapsto 3, y \mapsto \texttt{true} \right]
      \] 
      \[
        \downarrow
      \] 
      \[
        \Delta = \left[ x \mapsto \texttt{int}, y \mapsto \texttt{bool} \right]
      \] 
    \end{example}

  \item Regola 3:
    \[
      \mathcal{D}_{S_3}: \;
      \begin{prooftree}
        \hypo{\Delta \vdash e : \tau}
        \infer1{\Delta \vdash \texttt{const x:}\; \tau = e : \left[ x \mapsto \tau \right]}
      \end{prooftree}
    \] 
    I tipi \textbf{devono essere tutti uguali}

  \item Regola 4:
    \[
      \mathcal{D}_{S_4}: \;
      \begin{prooftree}
        \hypo{\Delta \vdash d_1 : \Delta_1}
        \hypo{\Delta[\Delta_1] \vdash d_2 : \Delta_2}
        \infer2{\Delta \vdash d_1 \texttt{ in } d_2 : \Delta_2}
      \end{prooftree}
    \] 

  \item Regola 5:
    \[
      \mathcal{D}_{S_5}: \;
      \begin{prooftree}
        \hypo{\Delta \vdash d_1 : \Delta_1}
        \hypo{\Delta[\Delta_1] \vdash d_2 : \Delta_2}
        \infer2{\Delta \vdash d_1; d_2 : \Delta_1[\Delta_2]}
      \end{prooftree}
    \]
\end{itemize}

\subsubsection{Semantica dinamica per le dichiarazioni}
L'ambiente statico generato dalla semantica statica è compatibile con l'ambiente dinamico.
La forma della regola di transizione per le dichiarazioni è la seguente:
\[
  \rho \vdash_{\Delta} d \to_d d'
\] 
rappresenta che \( d \) viene elaborata, con un passo di elaborazione, in \( d' \)
nell'ambiente dinamico \( \rho  \).

\vspace{1em}
\noindent
Gli stati sono dichiarazioni generati dalla grammatica o ambienti
\[
  \mathcal{D} \cup Env
\] 
Le regole per le dichiarazioni sono:
\begin{itemize}
  \item Regola 1: Assioma
    \[
      \mathcal{D}_1: \; \vdash \texttt{nil} \to \varnothing
    \] 
    \( d = \rho  \) è già terminale, non c'è nulla da elaborare

  \item Regola 2: Dichiarazione di una costante
    \[
      \mathcal{D}_2: \; \rho \vdash \texttt{const x:}\; \tau = k
      \to \left[ x \mapsto k \right]
    \] 

  \item Regola 3: Elaborazione di un'espressione
    \[
      \mathcal{D}_3: \;
      \begin{prooftree}
        \hypo{ \rho \vdash e \to_e e' }
        \infer1{ \rho \vdash \texttt{const x:}\; \tau = e \to_d 
        \texttt{const x:}\; \tau = e' }
      \end{prooftree}
    \] 

  \item Regola 4: Elaborazione di una composizione sequenziale
    \[
      \mathcal{D}_4: \;
      \begin{prooftree}
        \hypo{\rho \vdash d_1 \to_d d'_1}
        \infer1{\rho \vdash d_1; d_2 \to_d d'_1; d_2}
      \end{prooftree}
    \] 

  \item Regola 5: Elaborazione di una composizione sequenziale con un ambiente
    già elaborato
    \[
      \mathcal{D}_5: \;
      \begin{prooftree}
        \hypo{\rho[\rho_1] \vdash d_2 \to_d d'_2}
        \infer1{\rho \vdash \rho_1; d_2 \to_d \rho_1; d'_2}
      \end{prooftree}
    \]

  \item Regola 6: Assioma per la composizione privata, quando entrambi gli ambienti
    sono stati elaborati
    \[
      \mathcal{D}_6: \; \rho \vdash \rho_1; \rho_2 \to_d \rho_1[\rho_2]
    \] 

  \item Regola 7: Elaborazione di una composizione privata
    \[
      \mathcal{D}_7: \;
      \begin{prooftree}
        \hypo{\rho \vdash d_1 \to_d d'_1}
        \infer1{\rho \vdash d_1 \texttt{ in } d_2 \to_d d'_1 \texttt{ in } d_2}
      \end{prooftree}
    \]

  \item Regola 8: Elaborazione di una composizione privata con un ambiente già elaborato
    \[
      \mathcal{D}_8: \;
      \begin{prooftree}
        \hypo{\rho[\rho_1] \vdash d_2 \to_d d'_2}
        \infer1{\rho \vdash \rho_1 \texttt{ in } d_2 \to_d \rho_1 \texttt{ in } d'_2}
      \end{prooftree}
    \]

  \item Regola 9: Assioma per la composizione privata, quando entrambi gli ambienti
    sono stati elaborati
    \[
      \mathcal{D}_9: \; \rho \vdash \rho_1 \texttt{ in } \rho_2 \to_d \rho_2
    \]
\end{itemize}

\subsubsection{Elaborazione}
\begin{definition}[Funzione di elaborazione]
  La gunzione di elaborazione genera un ambiente dinamico a partire da una dichiarazione
  della grammatica:
  \[
    Elab: \mathcal{D} \to Env
  \] 
  \[
    \forall d \in \mathcal{D} \;.\; Elab(d) = \rho \iff \vdash d \to_d^* \rho
  \] 
  cioè l'elaborazione di \( d \) è l'ambiente \( \rho  \) se e solo se \( d \) viene
  elaborata in \( \rho  \) con un numero finito di passi di elaborazione.
\end{definition}

\begin{definition}[Equivalenza tra dichiarazioni]
  La relazione di equivalenza tra due dichiarazioni è definita come:
  \[
    \equiv \subset \mathcal{D} \times \mathcal{D}
  \] 
  \[
    \forall d_1, d_2 \in \mathcal{D} \;.\; d_1 \equiv d_2 \iff Elab(d_1) = Elab(d_2)
  \] 
  cioè due dichiarazioni sono equivalenti se e solo se la loro elaborazione genera
  lo stesso ambiente dinamico.
\end{definition}

\subsection{Comandi}
L'introduzione di \textbf{identificatori variabili} richiede l'introduzione del
concetto della memoria e degli strumenti per manipolarla, cioè i \textbf{comandi}.
Un classico assegnamento di una variabile è il seguente:
\[
  \underbrace{\texttt{x}}_{\text{Target}} \texttt{ := }
  \underbrace{\texttt{x}}_{\text{Sorgente}} \texttt{+1}
\] 
In questo caso si osserva che il ruolo della \texttt{x} cambia a seconda di dove
si trova:
\begin{itemize}
  \item Se si trova a sinistra è detto \textbf{L-value} (target) e il valore viene 
    usato in scrittura
  \item Se si trova a destra è detto \textbf{R-value} (sorgente) e il valore viene
    usato in lettura
\end{itemize}
Per gestire questo cambio di ruolo è necessario:
\begin{itemize}
  \item Gestire diversi oggetti con lo stesso nome (ad esempio nella ricorsione o nella
    definizione delle variabili)

  \item Gestire diversi nomi per lo stesso oggetto (ad esempio puntatori o aliasing)

  \item Gestire nomi e valori che cambiano durante l'esecuzione
\end{itemize}
È stato quindi introdotto un concetto astratto detto \textbf{locazione} che gestisce
questi aspetti.

\subsubsection{Locazione}
La locazione è un elemento per le variabili che viene inserito tra l'identificatore
e il valore o oggetto denotato. Fino ad ora abbiamo avuto soltanto un legame
identificatore-valore:
\[
  \texttt{id} \longleftrightarrow \texttt{valore/oggetto}
\] 
Con l'introduzione delle locazioni, il legame diventa:
\[
  \texttt{id} \stackrel{\text{Ambiente}}{\longleftrightarrow} \texttt{locazione}
  \stackrel{\text{Memoria}}{\longleftrightarrow} \texttt{valore/oggetto}
\] 
\begin{itemize}
  \item L'ambiente gestisce delle \textbf{trasformazioni reversibili}, quindi
    potenzialmente temporanee perchè possono essere sovrascritte da altre trasformazioni
    per poi essere recuperate in un secondo momento

  \item La memoria gestisce delle \textbf{trasformazioni irreversibili}, quindi
    permanenti perchè cambiare il valore in memoria significa perdere il valore precedente
    e sovrascriverlo con un nuovo valore. Nonostante il valore precedente possa essere
    riassegnato in un secondo momento, non è lo stesso valore di prima, ma è
    un nuovo valore che sovrascrive quello precedente
\end{itemize}

\begin{definition}[Memoria]
  La memoria tiene traccia delle trasformazioni effettuate dall'esecuzione di
  programmi
\end{definition}

\begin{definition}[Locazione]
  La locazione viene creata dalle dichiarazioni:
  \begin{itemize}
    \item Se la dichiarazione è esplicita, allora viene creata a tempo di compilazione
    \item Se la dichiarazione è implicita, allora viene creata dai comandi a tempo di
      esecuzione (nei linguatti dinamici/interpretati)
  \end{itemize}
  Le locazioni:
  \begin{itemize}
    \item rappresentano una posizione in memoria e ha un \textbf{tempo di vita},
      cioè un intervallo di tempo in cui è attiva. Il tempo di vita di una locazione
      dipende dal linguaggio di programmazione, solitamente coincide con la durata di tutto
      il programma, ma in alcuni linguaggi può essere limitato a un blocco di codice o
      alle allocazioni e deallocazioni dinamiche.

    \item Vengono usate per contenere/riferire oggetti che possono cambiare durante
      l'esecuzione

    \item Sono considerate illimitate

    \item Sono oggetti denotabili
  \end{itemize}
\end{definition}
Ci sono più tipi di legami:
\begin{itemize}
  \item \textbf{Identificatori costanti}: non usano la memoria perchè vengono associati
    a valori che non cambiano durante l'esecuzione. Sono quindi legami che
    \textbf{restano nell'ambiente dinamico}. Come notazione si usa:
    \[
      \texttt{const}
    \] 
    che crea un legame (bindin) in un ambiente.

  \item \textbf{Variabili} (nei linguaggi imperativi): sono identificate da due legami:
    \begin{itemize}
      \item Uno nell'ambiente che rappresenta la locazione
        \[
          x \to \square
        \] 
      \item Uno nella memoria che rappresenta il valore a cui la locazione fa
        riferimento che può anche cambiare
        \[
          \square \to 2
        \] 
    \end{itemize}
    Come notazione si usa:
    \[
      \texttt{var}
    \] 
    che crea un \textbf{legame} in un ambiente e una \textbf{associazione} in memoria:
    \[
      x \underbrace{\longrightarrow}_{\substack{\text{Legame}\\\text{in ambiente}}}
      \underbrace{\square}_{\text{Locazione}}
      \underbrace{\longrightarrow}_{\substack{\text{Associazione}\\\text{in memoria}}}
      2
    \] 
\end{itemize}

\vspace{1em}
\noindent
Nel nostro linguaggio abbiamo che:
\begin{itemize}
  \item I valori \textbf{esprimibili} sono elementi primitivi di \( Exp \):
    \[
      \texttt{B} \cup \texttt{N}
    \] 

  \item I valori \textbf{memorizzabili}, cioè che possono essere associati ad una
    locazione, sono:
    \[
      \texttt{B} \cup \texttt{N}
    \] 

  \item I valori \textbf{denotabili}, cioè che possono essere legati ad un identificatore,
    sono:
    \[
      \texttt{B} \cup \texttt{N} \cup \underbrace{L}_{\text{Locazioni}}
    \] 
\end{itemize}

\subsubsection{Implementazione delle variabili nelle dichiarazioni}
Modifichiamo le dichiarazioni per introdurre le variabili:
\[
  \begin{aligned}
    \mathcal{D}: \; D & \to  \texttt{ nil}  \\
                      & \mid \texttt{const x: } \tau = e  \\
                      & \mid \mathcolor{Blue}{\texttt{var x:}\; \tau = e} \\
                      & \mid D; D  \mid D \texttt{ in } D \\
                      & \mid \rho
  \end{aligned}
\] 
Questa sintassi dichiara una variabile di nome \texttt{x}, tipo \( \tau \) e valore
\textbf{iniziale} descritto da \( e \).
\begin{itemize}
  \item Gli identificatori liberi sono:
    \[
      FI(\texttt{var x:}\; \tau = e) = FI(e)
    \] 

  \item Gli identificatori definiti sono:
    \[
      DI(\texttt{var x:}\; \tau = e) = \left\{ x \right\}
    \]
\end{itemize}
Bisogna verificare, a livello statico, che non sia possibile modificare il valore
di una costante. Per fare ciò bisogna aggiungere i tipi per le locazioni:
\[
  DTyp = \left\{ \texttt{int}, \texttt{bool}, \mathcolor{Blue}{\texttt{intloc}},
  \mathcolor{Blue}{\texttt{boolloc}} \right\}
\] 
Ora bisogna definire le regole statiche per le dichiarazioni con variabili:
\begin{itemize}
  \item Regola 1:
    \[
      \begin{prooftree}
        \hypo{ \Delta \vdash e : \tau }
        \infer1{ \Delta \vdash \texttt{var x:}\; \tau = e : \left[ x \mapsto \tau\texttt{loc} \right] }
      \end{prooftree}
    \] 
\end{itemize}
